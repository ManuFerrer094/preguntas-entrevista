En "Cómo diseñarías formularios complejos en Vue para mantener validación, accesibilidad, rendimiento y contratos claros de `v-model`" se ve si piensas en contratos de campo y en experiencia de usuario a la vez.

## Qué evalúa el entrevistador
- Si separas estado de entrada, validación, accesibilidad y envío sin mezclarlo todo en un componente gigantesco.
- Si diseñas contratos de `v-model` y mensajes de error que sigan siendo mantenibles cuando el formulario crece.
- Si entiendes cómo proteger la experiencia del usuario ante errores, latencia y campos dependientes.

## Respuesta sólida
- Modela el formulario como flujo: captura, validación, transformación, envío y recuperación ante error.
- Haz explícitos los contratos de campo, los estados pendientes y la accesibilidad de feedback y focos.
- Cuando el formulario escala, reparte responsabilidades entre componentes de campo, reglas y composición de feature.

## Compromisos y errores comunes
- Mezclar validación visual con reglas de negocio termina generando componentes rígidos y muy difíciles de probar.
- Un formulario complejo sin contrato claro de `v-model` ni mensajes accesibles acaba rompiendo UX y mantenimiento a la vez.

## Ejemplo de código
```vue
<script setup lang="ts">
const model = defineModel<string>({ required: true });
const props = defineProps<{ error?: string; label: string }>();
</script>

<template>
  <label>
    <span>{{ props.label }}</span>
    <input v-model="model" :aria-invalid="Boolean(props.error)" />
  </label>
  <p v-if="props.error" role="alert">{{ props.error }}</p>
</template>
```

## Ejemplo o caso real
Un caso claro aparece en un formulario largo con pasos, campos dependientes y varios estados de error y guardado. Si "Cómo diseñarías formularios complejos en Vue para mantener validación, accesibilidad, rendimiento y contratos claros de `v-model`" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> Si el usuario no entiende el error o el equipo no entiende el flujo, el formulario está mal diseñado.
