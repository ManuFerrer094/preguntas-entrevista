Este caso mide si depuras con orden o si cambias piezas al azar hasta que el síntoma desaparece.

## Qué evalúa el entrevistador
- Si reduces el problema antes de corregirlo y sabes observar qué dispara cada cambio relevante.
- Si revisas orden de eventos, ciclo de vida, peticiones en vuelo y conservación de componentes con método.
- Si dejas una forma de prevenir la recaída después de arreglar el bug.

## Respuesta sólida
- Haz visible el flujo con logs, herramientas, tests o pasos de reproducción mínimos antes de tocar diseño.
- Aísla si el fallo viene de lectura, escritura, conservación de estado o asincronía compitiendo por el último valor.
- Corrige la causa y deja una prueba, una métrica o una simplificación que evite repetir la misma clase de error.

## Compromisos y errores comunes
- Cambiar varias cosas a la vez puede “arreglar” el síntoma pero deja intacta la causa real.
- Sin limpieza, cancelación o reglas claras de activación, los bugs reactivos vuelven con otra forma.

## Ejemplo de código
```ts
const store = useUsersStore();
const { users, selectedId, pending } = storeToRefs(store);

watch(selectedId, (id) => {
  if (id) void store.loadDetails(id);
});
```

## Ejemplo o caso real
Un caso claro aparece en una vista detalle que sigue leyendo una copia local mientras Pinia ya contiene el valor correcto. Si "La UI muestra datos viejos aunque el store cambió: ¿cómo localizarías el origen" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> Si no sé quién escribe el dato y cuándo, todavía no estoy listo para corregir el bug.
