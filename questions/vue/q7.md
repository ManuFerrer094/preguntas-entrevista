Esta pregunta comprueba si sabes modernizar por tramos útiles y no por una promesa vaga de limpieza total.

## Qué evalúa el entrevistador
- Si planteas evolución incremental, convivencia temporal y retirada ordenada de la capa antigua.
- Si eliges prioridades por dolor real, riesgo y frecuencia de cambio en lugar de modernizar por orgullo técnico.
- Si dejas visible cuándo una solución puente deja de ser temporal.

## Respuesta sólida
- Empieza por los puntos legacy que más frenan entregas, revisión o depuración.
- Introduce facades, composables puente o stores nuevos para convivir un tiempo con la parte antigua sin duplicar lógica de negocio.
- Acompaña la modernización con reglas para el código nuevo y criterios de retirada para el código heredado.

## Compromisos y errores comunes
- Cambiar sintaxis sin tocar los acoplamientos principales apenas mejora el sistema.
- Mantener dos modelos de trabajo sin fecha de salida hace permanente una deuda que nació como temporal.

## Ejemplo de código
```ts
export function useSessionFacade() {
  const legacyStore = useLegacyVuexStore();
  const session = useSessionStore();

  return {
    user: computed(() => session.user ?? legacyStore.state.session.user),
    login: session.login,
  };
}
```

## Ejemplo o caso real
Un caso claro aparece en un dashboard heredado con mixins, filtros globales y componentes grandes que siguen cambiando cada sprint. ## Frase corta de entrevista
> Cada puente temporal necesita fecha de salida o acabará siendo la nueva deuda estable.
