Aquí se valora si encuentras señales tempranas de acoplamiento y pérdida de reactividad antes de que se conviertan en incidencias.

## Qué evalúa el entrevistador
- Si detectas señales tempranas de acoplamiento, pérdida de reactividad y tipado inútil antes de que lleguen a producción.
- Si priorizas el flujo del dato y el contrato público por encima de detalles cosméticos.
- Si revisas la PR pensando en la siguiente persona que tendrá que extender esa funcionalidad.

## Respuesta sólida
- Lee primero fuentes de verdad, puntos de escritura y dependencias invisibles con store, router y DOM.
- Busca casts rápidos, copias locales innecesarias, watchers para derivación y contratos de componente poco claros.
- Pide cambios donde la PR haga más opaco el sistema, aunque la funcionalidad parezca pasar todos los casos felices.

## Compromisos y errores comunes
- Aprobar código que funciona hoy pero no explica su contrato suele trasladar la complejidad a la siguiente iteración.
- Centrarse solo en estilo deja pasar la mayoría de problemas caros de mantener en Vue.

## Ejemplo de código
```ts
const store = useUsersStore();
const { users } = store;

watch(users, () => {
  localCache.value = users;
}, { deep: true });
```

## Ejemplo o caso real
Un caso claro aparece en una PR grande donde la funcionalidad parece pasar, pero el flujo del dato ya se vuelve difícil de seguir. Si "En una PR Vue importante, ¿qué anti-patrones de reactividad, tipado y acoplamiento revisarías antes de aprobar" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> Reviso para el próximo cambio, no solo para el commit actual.
