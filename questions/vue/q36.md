Esta pregunta deja ver si persigues confianza útil o solo una batería de tests que obliga a tener miedo del refactor.

## Qué evalúa el entrevistador
- Si priorizas pruebas por comportamiento crítico y no por cobertura decorativa.
- Si sabes aislar dependencias sin dejar tests acoplados al detalle de implementación.
- Si distingues bien qué debería comprobar una prueba unitaria, una integración y una navegación completa.

## Respuesta sólida
- Testea comportamiento observable, contratos de datos y ramas de error que sí cambian riesgo.
- Mockea lo justo para mantener señal y velocidad, pero sin inventarte un sistema distinto al real.
- Revisa la suite como producto: tiempo de ejecución, legibilidad y utilidad de los fallos cuando algo se rompe.

## Compromisos y errores comunes
- Fijar estructura interna del componente hace que la suite grite por refactors sanos y calle ante bugs reales.
- Una mezcla sin criterio de mocks y montajes completos genera pruebas lentas y difíciles de confiar.

## Ejemplo de código
```ts
it("envía el formulario cuando los datos son válidos", async () => {
  const wrapper = mount(ProfileForm);
  await wrapper.get("input[name=\"email\"]").setValue("manu@example.com");
  await wrapper.get("form").trigger("submit.prevent");
  expect(api.saveProfile).toHaveBeenCalled();
});
```

## Ejemplo o caso real
Un caso claro aparece en una suite que se rompe por refactors visuales menores aunque la funcionalidad siga bien. ## Frase corta de entrevista
> No persigo cobertura; persigo confianza útil cuando hago cambios.
