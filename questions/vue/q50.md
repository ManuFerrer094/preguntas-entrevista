## Respuesta
- Formula la decisión a partir del problema actual, las opciones disponibles y un criterio concreto para elegir entre ellas.
- Propón una adopción pequeña y medible antes de imponer una convención a todo el código de Vue o Nuxt.
- Deja por escrito alcance, límites y cuándo revisar la decisión para que siga siendo útil cuando cambie el contexto.

## Puntos clave
- Una decisión técnica madura explica problema, opciones, criterio y consecuencias; no solo la herramienta favorita del momento.
- En equipos Vue/Nuxt suele ser mejor adoptar con pilotos pequeños y ejemplos reales que forzar una convención de golpe sobre todo el código.
- Documentar alcance y fecha de revisión evita que una buena decisión de hoy se convierta en un dogma descontextualizado mañana.

## Errores comunes
- Adoptar una práctica nueva sin explicar el dolor actual genera resistencia legítima y poca adhesión real.
- Si una decisión no define cuándo revisarla, puede volverse norma incluso cuando el contexto ya cambió.

## Ejemplo de código
```ts
type DecisionRecord = {
  title: string;
  context: string;
  chosenOption: string;
  consequences: string[];
};

const record: DecisionRecord = {
  title: "Adoptar Pinia para estado compartido",
  context: "Duplicidad de lógica entre pantallas y composables",
  chosenOption: "Pinia solo para flujos cruzados",
  consequences: ["menos acoplamiento", "migración incremental"],
};
```

## Ejemplo o caso real
Suele darse cuando el equipo quiere introducir Nuxt, Pinia o una convención nueva y necesita una razón compartida para hacerlo sin romper la velocidad de entrega.

## Idea clave
> Una decisión técnica buena explica por qué se toma, cuándo revisar y qué coste asume.
