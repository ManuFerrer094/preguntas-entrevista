Aquí se mide cómo conviertes una discusión sobre Vue o Nuxt en una decisión visible, reversible y entendible para el equipo.

## Qué evalúa el entrevistador
- Si conviertes una discusión técnica en una decisión con contexto, opciones y consecuencias entendibles para el equipo.
- Si piensas en adopción, documentación mínima y revisión futura en lugar de cerrar el debate por jerarquía.
- Si sabes diferenciar una convención ligera de una decisión que sí merece quedar registrada.

## Respuesta sólida
- Define el problema antes que la herramienta y deja claro qué criterio hará ganar a una opción sobre otra.
- Propón una adopción pequeña, observable y corregible si la hipótesis no se confirma en el código real.
- Documenta alcance, límites y señales de revisión para que la decisión no dependa de memoria oral.

## Compromisos y errores comunes
- Adoptar una práctica nueva sin explicar el dolor actual suele generar resistencia legítima.
- Una decisión sin límites claros termina convertida en dogma incluso cuando el contexto ha cambiado.

## Ejemplo de código
```ts
type DecisionRecord = {
  title: string;
  context: string;
  chosenOption: string;
  consequences: string[];
};

const record: DecisionRecord = {
  title: "Adoptar Pinia para estado compartido",
  context: "Duplicidad de lógica entre pantallas y composables",
  chosenOption: "Pinia solo para flujos cruzados",
  consequences: ["menos acoplamiento", "migración incremental"],
};
```

## Ejemplo o caso real
Un caso claro aparece en un equipo que debate si adoptar Nuxt, mover estado a Pinia o normalizar nuevas convenciones de componente. Si "Cómo tomarías y comunicarías decisiones arquitectónicas sobre Vue/Nuxt, Pinia o nuevas prácticas dentro del equipo" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> No tomo decisiones para ganar discusiones; las tomo para que el equipo cambie mejor.
