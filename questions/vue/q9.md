Aquí lo relevante en "Cuándo usarías `ref`, `reactive`, `shallowRef`, `shallowReactive` o `markRaw`" es si sabes decidir quién posee el dato, quién lo deriva y quién reacciona fuera del componente.

## Qué evalúa el entrevistador
- Si distingues fuente de verdad, estado derivado y efecto externo sin mezclar esos papeles.
- Si sabes cuándo un valor debe calcularse, observarse o simplemente almacenarse con una referencia clara.
- Si puedes explicar el coste de cada decisión reactiva antes de que aparezca el bug.

## Respuesta sólida
- Empieza nombrando qué dato manda y qué datos pueden derivarse de él.
- Usa los mecanismos reactivos para el papel correcto y revisa cualquier duplicidad que parezca “más cómoda”.
- Si hay efectos externos, añade limpieza, cancelación y reglas de escritura visibles.

## Compromisos y errores comunes
- Usar `watch` para derivar estado que cabe en `computed` multiplica acoplamientos y desajustes.
- Perder la referencia reactiva al copiar o destructurar datos suele explicar pantallas con información vieja.

## Ejemplo de código
```ts
const editor = shallowRef<Editor | null>(null);
const filters = reactive({ status: "open" });

editor.value = markRaw(createEditor());
filters.status = "closed";
```

## Ejemplo o caso real
Un caso claro aparece en una pantalla que mezcla estado de UI simple con instancias pesadas de librerías externas. Si "Cuándo usarías `ref`, `reactive`, `shallowRef`, `shallowReactive` o `markRaw`" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> Si no sé quién posee el dato, sé que acabaré persiguiendo bugs de reactividad.
