La calidad de la respuesta en "Qué criterio usarías para mover lógica de un componente a Pinia o dejarla local" depende de si conviertes el estado en una fuente de verdad clara y trazable.

## Qué evalúa el entrevistador
- Si sabes decidir qué estado debe ser local, compartido o remoto sin duplicarlo en varias capas.
- Si modelas acciones, lecturas y escrituras con una fuente de verdad clara.
- Si piensas en consistencia y trazabilidad antes de hablar de la librería.

## Respuesta sólida
- Declara quién posee el dato y qué piezas solo lo observan o lo transforman.
- Conserva en Pinia solo lo que de verdad cruza pantallas, sesiones o flujos relevantes.
- Para asincronía, define cómo cancelas, invalidas, rehaces o reviertes sin esconderlo en watchers sueltos.

## Compromisos y errores comunes
- Duplicar información entre formulario, URL, caché y store hace muy difícil saber qué valor es el bueno.
- Llevar a estado global una necesidad temporal o local vuelve más opaca la aplicación sin resolver el diseño.

## Ejemplo de código
```ts
export const useCartStore = defineStore("cart", {
  state: () => ({ items: [] as CartItem[], pending: false }),
  getters: {
    total: (state) => state.items.reduce((sum, item) => sum + item.price * item.qty, 0),
  },
  actions: {
    add(item: CartItem) {
      this.items.push(item);
    },
  },
});
```

## Ejemplo o caso real
Un caso claro aparece en un componente que ha crecido porque comparte selección y filtros con varias partes de la misma feature. Si "Qué criterio usarías para mover lógica de un componente a Pinia o dejarla local" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> El estado global es caro: solo lo pago cuando la aplicación realmente lo necesita.
