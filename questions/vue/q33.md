## Qué evalúa el entrevistador
- Si mides antes de optimizar y sabes distinguir coste de render, red, memoria o bundle.
- Si eliges la palanca de mejora que aporta más valor con menos complejidad adicional.
- Si puedes relacionar la optimización con una métrica visible para usuario o equipo.

## Respuesta sólida
- Reproduce el cuello de botella y decide si el problema está en render, carga, serialización o trabajo que nunca debió dispararse.
- Ataca primero las optimizaciones baratas que mantienen el código entendible: división de código, listas virtuales, carga diferida y cálculos derivados correctos.
- Deja siempre una forma de verificar la mejora para no convertir la optimización en fe.

## Compromisos y errores comunes
- Optimizar sin una señal clara suele mover complejidad sin tocar el punto caro de verdad.
- Memorizar o conservar componentes porque sí acaba siendo tan costoso como volver a renderizarlos.

## Ejemplo de código
```ts
const ReportsPanel = defineAsyncComponent({
  loader: () => import("./features/reports/ReportsPanel.vue"),
  delay: 200,
  timeout: 10000,
  suspensible: true,
});
```
