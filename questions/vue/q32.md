El **Server-Side Rendering (SSR)** en Vue permite renderizar los componentes en el servidor y enviar HTML completo al cliente, mejorando el SEO y el tiempo de primera renderización. **Nuxt.js** es el framework meta de Vue que implementa SSR con configuración mínima.

## SSR básico con Vue (manual)

```javascript
// server.js (Node.js)
import { createSSRApp } from 'vue';
import { renderToString } from 'vue/server-renderer';
import { createRouter } from './router';
import App from './App.vue';

// Cada request crea una nueva instancia (no compartir estado entre requests)
async function renderApp(url) {
  const app = createSSRApp(App);
  const router = createRouter();

  app.use(router);

  // Navegar a la URL del request
  await router.push(url);
  await router.isReady();

  // Renderizar a HTML
  const html = await renderToString(app);
  return `
    <!DOCTYPE html>
    <html>
      <head><title>Mi App</title></head>
      <body>
        <div id="app">${html}</div>
        <script src="/client.js"></script>
      </body>
    </html>
  `;
}

// En el cliente: hidratación (no re-renderiza, añade interactividad)
// client.js
import { createSSRApp } from 'vue';
import App from './App.vue';

const app = createSSRApp(App);
app.mount('#app');  // Hidratación: conecta Vue al HTML del servidor
```

## Nuxt.js — SSR con configuración cero

```bash
npx nuxi init mi-proyecto-nuxt
cd mi-proyecto-nuxt && npm install
```

```javascript
// nuxt.config.ts
export default defineNuxtConfig({
  // Modo de renderizado por defecto: SSR
  ssr: true,

  // Módulos disponibles
  modules: ['@nuxtjs/tailwindcss', '@pinia/nuxt'],

  // Configuración de runtime (disponible en servidor y cliente)
  runtimeConfig: {
    // Solo servidor (privado)
    apiSecretKey: process.env.API_SECRET_KEY,
    // Servidor y cliente (público)
    public: {
      apiBaseUrl: process.env.API_BASE_URL || 'http://localhost:3000'
    }
  }
});
```

## Obtención de datos en Nuxt

```vue
<!-- pages/productos/[id].vue -->
<script setup>
// useAsyncData: carga datos en el servidor en SSR, en cliente en SPA
const route = useRoute();
const { data: producto, pending, error } = await useAsyncData(
  `producto-${route.params.id}`,  // Clave única para caché
  () => $fetch(`/api/productos/${route.params.id}`)
);

// useFetch: shorthand de useAsyncData + $fetch
const { data: relacionados } = await useFetch(
  `/api/productos/${route.params.id}/relacionados`,
  {
    key: `relacionados-${route.params.id}`,
    // Solo en cliente (no bloquea SSR):
    lazy: true,
    default: () => []
  }
);

// SEO con useHead
useHead({
  title: () => producto.value?.nombre ?? 'Producto',
  meta: [
    { name: 'description', content: () => producto.value?.descripcion }
  ]
});
</script>

<template>
  <div>
    <div v-if="pending">Cargando...</div>
    <div v-else-if="error">Error: {{ error.message }}</div>
    <div v-else-if="producto">
      <h1>{{ producto.nombre }}</h1>
      <p>{{ producto.descripcion }}</p>
      <span>{{ producto.precio }}€</span>
    </div>
  </div>
</template>
```

## API Routes en Nuxt (server/api/)

```typescript
// server/api/productos/[id].get.ts
export default defineEventHandler(async (event) => {
  const id = getRouterParam(event, 'id');

  // Acceso a DB desde el servidor (no expuesto al cliente)
  const producto = await db.productos.findUnique({ where: { id } });

  if (!producto) {
    throw createError({ statusCode: 404, message: 'Producto no encontrado' });
  }

  return producto;
});
```

## Modos de renderizado en Nuxt

```javascript
// nuxt.config.ts
export default defineNuxtConfig({
  routeRules: {
    '/':                { prerender: true },    // SSG (estático)
    '/productos/**':    { ssr: true },          // SSR (dinámico)
    '/dashboard/**':    { ssr: false },         // SPA (solo cliente)
    '/api/**':          { cors: true },         // Solo API
    '/admin/**':        { redirect: '/login' }, // Redirección
  }
});
```

## Plugins de Nuxt (ejecutados en inicialización)

```typescript
// plugins/auth.ts
export default defineNuxtPlugin(async (nuxtApp) => {
  const authStore = useAuthStore();

  // Se ejecuta en el servidor y en el cliente
  if (import.meta.server) {
    // Solo en servidor: leer cookie de sesión
    const cookie = useCookie('session-token');
    if (cookie.value) {
      await authStore.cargarUsuario(cookie.value);
    }
  }

  // Añadir helper global
  return {
    provide: {
      auth: authStore
    }
  };
});
```

## Diferencia entre renderizados

| Modo        | Cuándo renderiza     | SEO  | JS requerido | Datos frescos |
|-------------|----------------------|------|--------------|---------------|
| SSR         | En cada request      | ✅   | Hidratación  | ✅ Siempre    |
| SSG         | En build             | ✅   | Hidratación  | Solo en build |
| SPA/CSR     | En el navegador      | ❌   | ✅ Necesario | ✅ Siempre    |
| ISR         | Build + revalidación | ✅   | Hidratación  | Cada X tiempo |

**Consejo de entrevista**: En SSR, el servidor renderiza HTML y el cliente lo "hidrata" (no re-renderiza, solo añade event listeners). El error más común en SSR es compartir estado entre requests —en Vue/Nuxt cada request debe crear nuevas instancias de app, router y stores. Otro punto importante: las variables de entorno privadas en Nuxt solo deben estar en `runtimeConfig` (sin el prefijo `public`), no en archivos `.vue` para no exponerlas al cliente.
