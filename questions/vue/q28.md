En "Qué revisarías al implementar i18n y carga diferida de traducciones en Vue/Nuxt" lo importante es que navegación, permisos y carga de datos sigan teniendo reglas entendibles cuando la app crece.

## Qué evalúa el entrevistador
- Si ordenas navegación, permisos, carga de datos y división de código como un sistema coherente.
- Si sabes cuándo una decisión pertenece al router y cuándo debería vivir en la pantalla o en el servidor.
- Si mantienes rutas predecibles incluso cuando la aplicación crece por áreas de negocio.

## Respuesta sólida
- Piensa el router como contrato de navegación, no como lugar donde esconder cualquier preocupación transversal.
- Usa guards, middleware o carga de datos según el momento en que necesitas decidir o bloquear la navegación.
- Mantén carga diferida y metadatos cerca de la feature que representan para que el mapa de rutas siga siendo entendible.

## Compromisos y errores comunes
- Poner demasiada lógica de negocio en guards y middleware vuelve opaca la navegación y complica depuración.
- Disparar la misma carga desde router, componente y store suele terminar en peticiones duplicadas y estados inciertos.

## Ejemplo de código
```ts
async function switchLocale(locale: string) {
  if (!i18n.global.availableLocales.includes(locale)) {
    const messages = await import(`./locales/${locale}.json`);
    i18n.global.setLocaleMessage(locale, messages.default);
  }

  i18n.global.locale.value = locale;
}
```

## Ejemplo o caso real
Un caso claro aparece en una aplicación multidioma con rutas localizadas y mensajes cargados bajo demanda. Si "Qué revisarías al implementar i18n y carga diferida de traducciones en Vue/Nuxt" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> El router debe explicar cómo se navega, no esconder por qué falla la pantalla.
