En "Cómo explicarías la reactividad de Vue a alguien que mezcla `ref`, `reactive`, `computed` y `watch` sin criterio" se ve rápido si entiendes la reactividad como un modelo mental o solo como un menú de APIs.

## Qué evalúa el entrevistador
- Si distingues fuente de verdad, estado derivado y efecto externo sin mezclar esos papeles.
- Si sabes cuándo un valor debe calcularse, observarse o simplemente almacenarse con una referencia clara.
- Si puedes explicar el coste de cada decisión reactiva antes de que aparezca el bug.

## Respuesta sólida
- Empieza nombrando qué dato manda y qué datos pueden derivarse de él.
- Usa los mecanismos reactivos para el papel correcto y revisa cualquier duplicidad que parezca “más cómoda”.
- Si hay efectos externos, añade limpieza, cancelación y reglas de escritura visibles.

## Compromisos y errores comunes
- Usar `watch` para derivar estado que cabe en `computed` multiplica acoplamientos y desajustes.
- Perder la referencia reactiva al copiar o destructurar datos suele explicar pantallas con información vieja.

## Ejemplo de código
```vue
<script setup lang="ts">
import { computed, ref } from "vue";

const query = ref("");
const products = ref(["Vue", "Pinia", "Vitest"]);
const filtered = computed(() =>
  products.value.filter((item) => item.toLowerCase().includes(query.value.toLowerCase())),
);
</script>
```

## Ejemplo o caso real
Un caso claro aparece en una búsqueda con filtros, paginación y datos remotos donde nadie tiene claro qué valor manda. Si "Cómo explicarías la reactividad de Vue a alguien que mezcla `ref`, `reactive`, `computed` y `watch` sin criterio" está mal resuelto, el siguiente cambio pequeño acaba tocando más partes de las necesarias o dejando un comportamiento difícil de depurar.

## Frase corta de entrevista
> En Vue no mezclo derivación y efecto: primero calculo, luego sincronizo.
