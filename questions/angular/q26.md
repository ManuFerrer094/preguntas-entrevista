---
title: ¿Cuál es la diferencia entre constructor y ngOnInit?
difficulty: medium

---

Una de las dudas más frecuentes en entrevistas Angular es cuándo usar el **constructor** versus **`ngOnInit`**. Aunque ambos se ejecutan al crear el componente, tienen propósitos muy diferentes.

## El constructor

El **constructor** es un método estándar de TypeScript/JavaScript que se ejecuta cuando la clase es instanciada. En Angular, se ejecuta **antes** de que el framework haya procesado los `@Input()` del componente.

**Usos apropiados del constructor:**
- Inyección de dependencias (DI).
- Inicialización de propiedades simples con valores por defecto.
- Asignación de referencias de servicios.

```typescript
@Component({
  selector: 'app-perfil',
  standalone: true,
  template: `<p>{{ usuario?.nombre }}</p>`
})
export class PerfilComponent implements OnInit {
  usuario: Usuario | null = null;

  // ✅ Constructor: solo para inyección de dependencias
  constructor(
    private userService: UserService,
    private router: Router
  ) {
    // ❌ Incorrecto: this.userId aún no está disponible
    // this.userService.getUser(this.userId).subscribe(...)
  }

  @Input() userId!: number;

  // ✅ ngOnInit: lógica de inicialización, @Input() ya disponibles
  ngOnInit(): void {
    this.userService.getUser(this.userId).subscribe(user => {
      this.usuario = user;
    });
  }
}
```

## ngOnInit

`ngOnInit` es un **lifecycle hook** de Angular que se ejecuta **después** de que Angular ha inicializado todas las propiedades con binding de datos (`@Input()`). Es el lugar correcto para:

- Llamadas a APIs o servicios HTTP.
- Lógica que depende de los valores de `@Input()`.
- Suscripciones a Observables.
- Inicialización compleja del estado del componente.

## Tabla comparativa

| Característica             | `constructor`                  | `ngOnInit`                         |
|----------------------------|--------------------------------|------------------------------------|
| Ejecutado por              | TypeScript / JavaScript        | Angular framework                  |
| Momento de ejecución       | Al instanciar la clase         | Tras inicializar los `@Input()`    |
| `@Input()` disponibles     | ❌ No                          | ✅ Sí                              |
| Uso principal              | Inyección de dependencias      | Lógica de inicialización           |
| Llamadas HTTP              | ❌ No recomendado              | ✅ Recomendado                     |
| Acceso a DOM / ViewChild   | ❌ No                          | ❌ Tampoco (usar ngAfterViewInit)  |

## Ejemplo completo

```typescript
@Component({
  selector: 'app-detalle-producto',
  standalone: true,
  imports: [CommonModule, AsyncPipe],
  template: `
    @if (producto$ | async; as producto) {
      <h2>{{ producto.nombre }}</h2>
      <p>Precio: {{ producto.precio | currency:'EUR' }}</p>
    }
  `
})
export class DetalleProductoComponent implements OnInit, OnDestroy {
  @Input() productoId!: string;

  producto$!: Observable<Producto>;
  private destroy$ = new Subject<void>();

  // Solo DI en el constructor
  constructor(private productoService: ProductoService) {}

  // Lógica de negocio en ngOnInit
  ngOnInit(): void {
    this.producto$ = this.productoService.getProducto(this.productoId).pipe(
      takeUntil(this.destroy$)
    );
  }

  ngOnDestroy(): void {
    this.destroy$.next();
    this.destroy$.complete();
  }
}
```

## ¿Y los signals?

Con Angular Signals (v17+), la inicialización basada en efectos puede moverse fuera del ciclo de vida, pero `ngOnInit` sigue siendo válido y ampliamente utilizado:

```typescript
export class ProductoComponent {
  productoId = input.required<string>(); // Signal input

  // effect() se puede usar en el constructor con signals
  constructor(private productoService: ProductoService) {
    effect(() => {
      // Se re-ejecuta automáticamente cuando cambia productoId
      console.log('ID cambió:', this.productoId());
    });
  }
}
```

**Consejo de entrevista**: La regla de oro es: **constructor solo para DI, ngOnInit para lógica**. Si ves código que hace peticiones HTTP en el constructor, es una mala práctica. Además, el testing se simplifica cuando la lógica está en `ngOnInit`, ya que puedes controlar cuándo se inicializa el componente en los tests con `fixture.detectChanges()`.
