**`ngOnChanges`** es un lifecycle hook de Angular que se ejecuta cada vez que un `@Input()` del componente recibe un nuevo valor. Es el primer hook del ciclo de vida en ejecutarse (antes de `ngOnInit`), y proporciona acceso tanto al valor anterior como al nuevo mediante el objeto `SimpleChanges`.

## Cuándo se ejecuta

1. **Antes de `ngOnInit`**: si el componente tiene `@Input()` con valores, `ngOnChanges` se ejecuta primero.
2. **En cada cambio de referencia** de un `@Input()`: cuando el padre actualiza el valor enlazado.

## Implementación

```typescript
import { Component, Input, OnChanges, SimpleChanges } from '@angular/core';

@Component({
  selector: 'app-detalle-producto',
  standalone: true,
  template: `
    <h2>{{ producto?.nombre }}</h2>
    <p>Stock: {{ stockCalculado }}</p>
  `
})
export class DetalleProductoComponent implements OnChanges {
  @Input() productoId!: number;
  @Input() cantidad: number = 1;

  producto: Producto | null = null;
  stockCalculado = 0;

  constructor(private productoService: ProductoService) {}

  ngOnChanges(changes: SimpleChanges): void {
    // changes es un objeto donde cada key es el nombre del @Input() que cambió

    if (changes['productoId']) {
      const { previousValue, currentValue, firstChange } = changes['productoId'];

      console.log('ID anterior:', previousValue);
      console.log('ID nuevo:', currentValue);
      console.log('¿Es el primer cambio?', firstChange);

      if (!firstChange && previousValue !== currentValue) {
        // Solo recargar si realmente cambió (no en la primera inicialización)
        this.cargarProducto(currentValue);
      }
    }

    if (changes['cantidad']) {
      this.stockCalculado = (this.producto?.stock ?? 0) - changes['cantidad'].currentValue;
    }
  }

  private cargarProducto(id: number): void {
    this.productoService.getProducto(id)
      .subscribe(p => this.producto = p);
  }
}
```

## El objeto SimpleChanges

```typescript
interface SimpleChanges {
  [propName: string]: SimpleChange;
}

interface SimpleChange {
  previousValue: any;  // Valor anterior
  currentValue: any;   // Valor nuevo
  firstChange: boolean; // true solo en la primera ejecución
  isFirstChange(): boolean; // Método equivalente a firstChange
}
```

## ngOnChanges vs setter en @Input()

Ambos permiten reaccionar a cambios de `@Input()`. ¿Cuándo usar cada uno?

```typescript
// ✅ Setter: para reaccionar al cambio de UN solo Input con lógica simple
@Input() set nombre(valor: string) {
  this._nombre = valor.trim().toUpperCase();
  this.nombreProcesado = `${this._nombre} (procesado)`;
}
private _nombre = '';
nombreProcesado = '';

// ✅ ngOnChanges: cuando necesitas comparar valores anteriores/actuales
// o cuando múltiples @Input() cambian y necesitas reaccionar juntos
ngOnChanges(changes: SimpleChanges): void {
  if (changes['filtro'] || changes['paginacion']) {
    // Recargar datos cuando cambia el filtro O la paginación
    this.cargarDatos(this.filtro, this.paginacion);
  }
}
```

## Limitación importante: no detecta mutaciones internas

`ngOnChanges` solo se dispara cuando **cambia la referencia** del objeto, no cuando se mutan sus propiedades internas:

```typescript
// El padre hace:
this.configuracion.opciones.push('nueva'); // ❌ ngOnChanges NO se dispara
// La referencia de 'configuracion' no cambió

this.configuracion = { ...this.configuracion, opciones: [...this.configuracion.opciones, 'nueva'] }; // ✅ SÍ se dispara
// La referencia de 'configuracion' sí cambió (nuevo objeto)
```

## Compatibilidad con Signals: `input()` + `effect()`

En Angular 17.1+, la nueva API `input()` ofrece una alternativa más reactiva:

```typescript
import { Component, input, effect, computed } from '@angular/core';

@Component({
  selector: 'app-detalle-moderno',
  standalone: true,
  template: `
    <p>ID: {{ productoId() }}</p>
    <p>Stock: {{ stockCalculado() }}</p>
  `
})
export class DetalleModernoComponent {
  productoId = input.required<number>();
  cantidad = input<number>(1);

  // effect() reacciona automáticamente cuando cambian los signals
  constructor(private productoService: ProductoService) {
    effect(() => {
      const id = this.productoId();
      // Se re-ejecuta cada vez que productoId cambia
      this.productoService.getProducto(id).subscribe(/* ... */);
    });
  }

  // computed() para valores derivados
  stockCalculado = computed(() => {
    // Se recalcula cuando cambian productoId o cantidad
    return 100 - this.cantidad();
  });
}
```

## Cuándo usar ngOnChanges

- Cuando necesitas **comparar** el valor anterior y el nuevo de un `@Input()`.
- Cuando varios `@Input()` cambian y debes ejecutar lógica que considera todos.
- Cuando el cambio de un `@Input()` requiere una petición HTTP o lógica asíncrona.
- Para implementaciones compatibles con las versiones actuales sin migrar a Signals.

**Consejo de entrevista**: Un error común es olvidar verificar `firstChange` en `ngOnChanges`. Sin esta comprobación, la lógica puede ejecutarse con `previousValue === undefined` en la primera inicialización, causando comportamientos inesperados. La práctica recomendada es siempre verificar `changes['miInput']?.firstChange` antes de acceder a `previousValue`. En proyectos nuevos con Angular 17+, considera usar `input()` con `effect()` como alternativa más reactiva y elegante.
