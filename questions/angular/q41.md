**XSS (Cross-Site Scripting)** es uno de los ataques web más comunes, en el que un atacante inyecta código malicioso (generalmente JavaScript) en una página web. Angular tiene mecanismos integrados para prevenirlo **por defecto**, sin requerir configuración adicional.

## Sanitización automática

Angular trata todos los valores como **no confiables** por defecto y los sanitiza automáticamente antes de insertarlos en el DOM.

```typescript
@Component({
  template: `
    <!-- ✅ SEGURO: Angular escapa el HTML automáticamente -->
    <p>{{ contenidoMalicioso }}</p>
    <!-- Si contenidoMalicioso = '<script>alert("XSS")</script>'
         Angular lo renderiza como texto literal, no como código -->
  `
})
export class EjemploComponent {
  contenidoMalicioso = '<script>alert("XSS")</script>';
  // Se muestra como texto: <script>alert("XSS")</script>
}
```

## Contextos de sanitización

Angular sanitiza los valores según el **contexto** donde se usan:

| Contexto       | Ejemplo                           | Acción de Angular                          |
|----------------|-----------------------------------|--------------------------------------------|
| HTML           | `[innerHTML]="valor"`             | Elimina tags peligrosos (script, on*)      |
| Style          | `[style]="valor"`                 | Elimina expresiones CSS peligrosas         |
| URL            | `[href]="url"`, `[src]="url"`     | Valida protocolo (bloquea `javascript:`)   |
| ResourceURL    | `[src]` en `<iframe>`, `<script>` | Bloquea URLs no confiables                 |
| Script         | Dinámico en `<script>`            | Siempre bloqueado                          |

```html
<!-- ✅ Angular sanitiza [innerHTML]: elimina <script> pero permite <b>, <p>, etc. -->
<div [innerHTML]="htmlConfiable"></div>

<!-- ⚠️ Angular bloquea URLs con protocolo javascript: -->
<a [href]="urlDinamica">Enlace</a>
```

## DomSanitizer: confiar en contenido explícitamente

A veces necesitas insertar HTML, URLs o estilos que Angular sanitizaría. Para eso existe `DomSanitizer`:

```typescript
import { DomSanitizer, SafeHtml, SafeUrl } from '@angular/platform-browser';

@Component({
  template: `
    <div [innerHTML]="htmlSeguro"></div>
    <iframe [src]="urlSegura"></iframe>
  `
})
export class ContenidoComponent {
  htmlSeguro: SafeHtml;
  urlSegura: SafeUrl;

  constructor(private sanitizer: DomSanitizer) {
    // ⚠️ Solo usar cuando el contenido viene de una fuente CONFIABLE
    this.htmlSeguro = this.sanitizer.bypassSecurityTrustHtml(
      '<strong>Contenido de confianza</strong>'
    );

    this.urlSegura = this.sanitizer.bypassSecurityTrustResourceUrl(
      'https://www.youtube.com/embed/VIDEO_ID'
    );
  }
}
```

### Métodos de DomSanitizer

```typescript
sanitizer.bypassSecurityTrustHtml(value)         // Para [innerHTML]
sanitizer.bypassSecurityTrustStyle(value)         // Para [style]
sanitizer.bypassSecurityTrustUrl(value)           // Para [href], [src]
sanitizer.bypassSecurityTrustResourceUrl(value)   // Para src de iframe, script
sanitizer.bypassSecurityTrustScript(value)        // Para contenido de script
```

## ⛔ Advertencia: nunca usar bypassSecurityTrust* con datos del usuario

```typescript
// ❌ PELIGROSO: contenido del usuario sin validar
this.htmlSeguro = this.sanitizer.bypassSecurityTrustHtml(
  inputDelUsuario // Nunca hagas esto
);

// ✅ SEGURO: solo para contenido que controlas completamente
this.htmlSeguro = this.sanitizer.bypassSecurityTrustHtml(
  CONSTANTE_HTML_INTERNA
);
```

## Content Security Policy (CSP)

Además de la sanitización de Angular, se recomienda configurar una **CSP** en el servidor:

```
# Cabecera HTTP en el servidor
Content-Security-Policy: default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.ejemplo.com;
```

## Otras buenas prácticas de seguridad en Angular

1. **No usar `eval()`** ni `new Function()` con datos del usuario.
2. **Validar entradas** tanto en cliente como en servidor.
3. **Usar HttpClient** (que usa XHR con protecciones XSRF integradas) en lugar de `fetch` directamente.
4. **XSRF/CSRF**: Angular incluye protección automática con `HttpClientXsrfModule`.
5. **Actualizar dependencias** regularmente con `ng update`.

```typescript
// XSRF automático con HttpClient (Angular lo gestiona automáticamente)
// Solo necesitas el módulo si quieres personalizar el nombre del cookie/header
import { provideHttpClient, withXsrfConfiguration } from '@angular/common/http';

provideHttpClient(
  withXsrfConfiguration({
    cookieName: 'XSRF-TOKEN',
    headerName: 'X-XSRF-TOKEN'
  })
)
```

**Consejo de entrevista**: El punto clave es que Angular previene XSS **por defecto** sanitizando automáticamente todos los valores enlazados al DOM. El riesgo aparece cuando se usa `bypassSecurityTrustHtml` (u otros métodos similares) con contenido no controlado. Si un entrevistador pregunta sobre `[innerHTML]`, menciona que Angular lo sanitiza, eliminando elementos como `<script>` y atributos `onerror`, pero no lo hace completamente invulnerable; la defensa en profundidad con CSP es igualmente importante.
