---
title: ¿Cómo se maneja el estado en aplicaciones Angular?
difficulty: medium
tags: [angular, state-management, signals, rxjs]
---

El manejo del estado en Angular ha evolucionado significativamente. No existe una única solución; la elección depende del tamaño y complejidad de la aplicación. Las principales opciones van desde servicios simples hasta soluciones completas como NgRx.

**1. Servicios con Signals (recomendado para Angular 17+):**

La forma más moderna y sencilla de manejar estado local o compartido entre componentes:

```typescript
@Injectable({ providedIn: 'root' })
export class CartStore {
  private _items = signal<CartItem[]>([]);
  private _loading = signal(false);

  // Estado público de solo lectura
  readonly items = this._items.asReadonly();
  readonly loading = this._loading.asReadonly();
  readonly totalItems = computed(() => this._items().reduce((sum, item) => sum + item.quantity, 0));
  readonly totalPrice = computed(() => this._items().reduce((sum, item) => sum + item.price * item.quantity, 0));

  addItem(product: Product): void {
    this._items.update(items => {
      const existing = items.find(i => i.id === product.id);
      if (existing) {
        return items.map(i => i.id === product.id ? { ...i, quantity: i.quantity + 1 } : i);
      }
      return [...items, { ...product, quantity: 1 }];
    });
  }

  removeItem(id: string): void {
    this._items.update(items => items.filter(i => i.id !== id));
  }
}
```

**2. Servicios con BehaviorSubject (RxJS, Angular 8+):**

Útil cuando se necesita integración con otros observables:

```typescript
@Injectable({ providedIn: 'root' })
export class UserStateService {
  private userSubject = new BehaviorSubject<User | null>(null);
  user$ = this.userSubject.asObservable();

  setUser(user: User) { this.userSubject.next(user); }
  clearUser() { this.userSubject.next(null); }
  getUser() { return this.userSubject.getValue(); }
}
```

**3. NgRx Store (para aplicaciones grandes):**

Para aplicaciones enterprise con estado complejo, múltiples fuentes de datos y necesidad de debugging avanzado. Ver pregunta dedicada a NgRx para más detalles.

**4. Signals Store de NgRx (@ngrx/signals):**

Una nueva librería que combina la simplicidad de los signals con la estructura de NgRx:

```typescript
import { signalStore, withState, withComputed, withMethods } from '@ngrx/signals';

export const ProductsStore = signalStore(
  { providedIn: 'root' },
  withState({ products: [] as Product[], loading: false }),
  withComputed(({ products }) => ({
    totalCount: computed(() => products().length)
  })),
  withMethods((store, productService = inject(ProductService)) => ({
    async loadProducts() {
      patchState(store, { loading: true });
      const products = await firstValueFrom(productService.getAll());
      patchState(store, { products, loading: false });
    }
  }))
);
```

**¿Cuándo usar qué?**

| Escenario | Solución recomendada |
|---|---|
| Estado local de componente | `signal()` local |
| Estado compartido simple | Servicio con Signals |
| Flujos reactivos complejos | BehaviorSubject + RxJS |
| App grande, equipo amplio | NgRx Store |
| App mediana con estructura | @ngrx/signals |

**Consejo de entrevista**: En 2024-2025, la respuesta moderna es usar Signals para estado local y servicios con signals para estado compartido. NgRx sigue siendo relevante para aplicaciones enterprise, pero ya no es la primera opción para todo.
