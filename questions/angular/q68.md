---
title: ¿Qué son los Shared Modules en Angular?

Un **Shared Module** (módulo compartido) es un `@NgModule` que centraliza y exporta componentes, directivas, pipes y otros módulos que se usan en múltiples partes de la aplicación. Evita importar los mismos elementos en cada feature module.

## Estructura típica de un Shared Module

```typescript
// shared/shared.module.ts
import { NgModule } from '@angular/core';
import { CommonModule } from '@angular/common';
import { ReactiveFormsModule, FormsModule } from '@angular/forms';
import { RouterModule } from '@angular/router';

// Componentes compartidos
import { SpinnerComponent } from './components/spinner/spinner.component';
import { AlertaComponent } from './components/alerta/alerta.component';
import { TarjetaComponent } from './components/tarjeta/tarjeta.component';
import { PaginadorComponent } from './components/paginador/paginador.component';

// Directivas compartidas
import { ResaltarDirective } from './directives/resaltar.directive';
import { PermisoDirective } from './directives/permiso.directive';
import { DebounceClickDirective } from './directives/debounce-click.directive';

// Pipes compartidos
import { TruncarPipe } from './pipes/truncar.pipe';
import { TiempoRelativoPipe } from './pipes/tiempo-relativo.pipe';

@NgModule({
  declarations: [
    // Solo componentes, directivas y pipes NO standalone
    SpinnerComponent,
    AlertaComponent,
    TarjetaComponent,
    PaginadorComponent,
    ResaltarDirective,
    PermisoDirective,
    DebounceClickDirective,
    TruncarPipe,
    TiempoRelativoPipe
  ],
  imports: [
    // Módulos externos que también re-exportamos
    CommonModule,
    ReactiveFormsModule,
    FormsModule,
    RouterModule
  ],
  exports: [
    // ✅ Exportar todo para que los módulos que importen SharedModule
    // tengan acceso a todos estos elementos
    CommonModule,
    ReactiveFormsModule,
    FormsModule,
    RouterModule,
    SpinnerComponent,
    AlertaComponent,
    TarjetaComponent,
    PaginadorComponent,
    ResaltarDirective,
    PermisoDirective,
    DebounceClickDirective,
    TruncarPipe,
    TiempoRelativoPipe
  ]
  // ❌ NO añadir providers aquí (ver reglas abajo)
})
export class SharedModule {}
```

## Uso en feature modules

```typescript
// features/productos/productos.module.ts
@NgModule({
  declarations: [ProductosListComponent, ProductoCardComponent],
  imports: [
    SharedModule,       // ← Un solo import da acceso a todo
    ProductosRoutingModule
  ]
})
export class ProductosModule {}
```

## ⚠️ Regla crítica: no proveer servicios singleton en Shared Modules

```typescript
// ❌ MAL: si dos módulos importan SharedModule, cada uno creará su propia
// instancia del servicio, rompiendo el patrón singleton
@NgModule({
  providers: [
    CarritoService,  // ❌ NUNCA esto en un SharedModule
    AuthService      // ❌ NUNCA esto en un SharedModule
  ]
})
export class SharedModule {}

// ✅ BIEN: los servicios singleton van en el propio servicio
@Injectable({ providedIn: 'root' }) // ← Singleton garantizado
export class CarritoService {}

// ✅ BIEN: o en CoreModule con forRoot()
@NgModule({
  providers: [ServicioSingleton]
})
export class CoreModule {
  static forRoot(): ModuleWithProviders<CoreModule> {
    return { ngModule: CoreModule, providers: [ServicioSingleton] };
  }
}
```

## Shared Module en la era de Standalone Components (Angular 17+)

Con los Standalone Components, la necesidad de Shared Modules disminuye significativamente. Cada componente standalone importa directamente lo que necesita:

```typescript
// ✅ Standalone Component: importa directamente sin SharedModule
@Component({
  selector: 'app-productos-list',
  standalone: true,
  imports: [
    CommonModule,         // O solo los pipes/directivas individuales
    ReactiveFormsModule,
    TruncarPipe,          // Pipe standalone
    TarjetaComponent,     // Componente standalone
    SpinnerComponent,     // Componente standalone
    ResaltarDirective,    // Directiva standalone
  ],
  template: `...`
})
export class ProductosListComponent {}
```

Sin embargo, se puede crear un "barrel" de exports para simplificar las importaciones:

```typescript
// shared/index.ts — Barrel file
export { SpinnerComponent } from './components/spinner/spinner.component';
export { AlertaComponent } from './components/alerta/alerta.component';
export { TarjetaComponent } from './components/tarjeta/tarjeta.component';
export { TruncarPipe } from './pipes/truncar.pipe';
export { ResaltarDirective } from './directives/resaltar.directive';

// Importación simplificada en componentes standalone
import { SpinnerComponent, TruncarPipe, TarjetaComponent } from '../shared';
```

## Qué incluir vs qué no incluir en un Shared Module

| ✅ Incluir                                        | ❌ No incluir                              |
|---------------------------------------------------|--------------------------------------------|
| Componentes reutilizables (botones, cards, etc.)  | Servicios singleton (usar `providedIn: root`)|
| Directivas genéricas                              | Componentes específicos de un feature      |
| Pipes de transformación                           | Módulos de feature completos               |
| CommonModule, FormsModule, ReactiveFormsModule    | BrowserModule (solo en AppModule)          |
| RouterModule (para routerLink, routerLinkActive)  | Componentes de páginas/vistas             |

**Consejo de entrevista**: El error más común con SharedModule es incluir servicios en `providers`. Esto crea múltiples instancias del servicio (una por módulo que importe SharedModule), rompiendo el patrón singleton. La solución es `providedIn: 'root'` en el `@Injectable` del servicio. En proyectos Angular 17+ con standalone components, los SharedModules se vuelven innecesarios; en su lugar se usan "barrel files" para agrupar exports relacionados.
