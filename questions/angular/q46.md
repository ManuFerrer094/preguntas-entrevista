Un **NgModule** es una clase decorada con `@NgModule` que actúa como **contenedor lógico** para un conjunto de componentes, directivas, pipes y servicios relacionados. Fue el mecanismo principal de organización de Angular hasta que los **Standalone Components** (Angular 14+) redujeron su necesidad.

## Estructura de un NgModule

```typescript
import { NgModule } from '@angular/core';
import { BrowserModule } from '@angular/platform-browser';
import { HttpClientModule } from '@angular/common/http';
import { FormsModule, ReactiveFormsModule } from '@angular/forms';

@NgModule({
  declarations: [
    // Componentes, directivas y pipes que pertenecen a ESTE módulo
    AppComponent,
    HeaderComponent,
    FooterComponent,
    TruncarPipe,
    ResaltarDirective
  ],
  imports: [
    // Otros módulos cuyas exportaciones necesitamos en este módulo
    BrowserModule,
    HttpClientModule,
    FormsModule,
    ReactiveFormsModule,
    UsuariosModule,       // Módulo de feature
    SharedModule          // Módulo compartido
  ],
  exports: [
    // Lo que este módulo expone para que otros módulos lo usen
    HeaderComponent,
    FooterComponent,
    TruncarPipe
  ],
  providers: [
    // Servicios disponibles en el inyector de este módulo
    // (preferible usar providedIn: 'root' en los servicios directamente)
    { provide: APP_CONFIG, useValue: appConfig }
  ],
  bootstrap: [
    // Solo en AppModule: componente raíz de la aplicación
    AppComponent
  ]
})
export class AppModule {}
```

## Las cuatro propiedades principales

| Propiedad      | Descripción                                                               |
|----------------|---------------------------------------------------------------------------|
| `declarations` | Componentes, directivas y pipes que pertenecen a este módulo              |
| `imports`      | Otros módulos de cuyas exportaciones se necesita en las plantillas        |
| `exports`      | Subconjunto de `declarations` e `imports` accesibles para otros módulos  |
| `providers`    | Servicios que se registran en el inyector del módulo                      |

## Módulos integrados de Angular

```typescript
import { BrowserModule } from '@angular/platform-browser';     // Solo en AppModule
import { CommonModule } from '@angular/common';                // En feature modules
import { FormsModule } from '@angular/forms';                  // Template-driven forms
import { ReactiveFormsModule } from '@angular/forms';          // Reactive forms
import { HttpClientModule } from '@angular/common/http';       // HTTP (antiguo)
import { RouterModule } from '@angular/router';                // Routing
```

## AppModule: el módulo raíz

```typescript
// app.module.ts (arquitectura clásica)
@NgModule({
  declarations: [AppComponent],
  imports: [
    BrowserModule,
    AppRoutingModule,
    CoreModule,        // Servicios singleton
    SharedModule       // Componentes/pipes compartidos
  ],
  bootstrap: [AppComponent]
})
export class AppModule {}
```

## NgModules vs Standalone Components (Angular 14+)

Desde Angular 14, los **Standalone Components** permiten declarar componentes sin pertenecer a ningún NgModule. Desde Angular 17, son el **modo por defecto** del CLI.

```typescript
// ❌ Con NgModule (clásico)
@NgModule({
  declarations: [UserCardComponent],
  imports: [CommonModule],
  exports: [UserCardComponent]
})
export class UserModule {}

// ✅ Standalone Component (moderno, Angular 17+)
@Component({
  selector: 'app-user-card',
  standalone: true,
  imports: [CommonModule],  // Las dependencias van aquí directamente
  template: `...`
})
export class UserCardComponent {}
```

## Bootstrap sin NgModule (Angular 17+)

```typescript
// main.ts moderno (sin AppModule)
import { bootstrapApplication } from '@angular/platform-browser';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';

bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes),
    provideHttpClient(),
    provideAnimations()
  ]
});
```

## ¿Cuándo usar NgModules todavía?

Aunque los Standalone Components son el futuro, los NgModules siguen siendo relevantes:
- **Proyectos heredados** (legacy) con arquitectura basada en módulos.
- **Librerías** que necesitan exportar múltiples componentes como un conjunto.
- **Lazy loading** de feature modules (aunque ahora también se puede hacer con `loadComponent`).

**Consejo de entrevista**: En proyectos modernos Angular 17+, los NgModules ya no son necesarios gracias a los Standalone Components. Sin embargo, es importante conocerlos porque la mayoría de los proyectos en producción hoy en día aún usan la arquitectura basada en módulos. El concepto de `imports`, `exports`, `declarations` y `providers` se entiende mejor si se comprende bien el sistema de NgModules, incluso aunque en el día a día trabajes con standalone.
