---
title: ¿Qué es trackBy en ngFor y por qué es importante para el rendimiento?
difficulty: medium
tags: [angular, ngFor, trackBy, performance]
---

**`trackBy`** es una función que se le proporciona a `*ngFor` para que Angular pueda identificar de forma única cada elemento de la lista. Esto permite que Angular **reutilice los nodos DOM existentes** en lugar de destruirlos y recrearlos cuando la lista cambia.

## El problema sin trackBy

Cuando un array se actualiza y Angular no puede distinguir qué elementos cambiaron, destruye y recrea **todos** los elementos del DOM:

```typescript
@Component({
  standalone: true,
  imports: [CommonModule],
  template: `
    <!-- ❌ Sin trackBy: Angular recrea TODOS los <li> cuando cambia 'usuarios' -->
    <ul>
      <li *ngFor="let usuario of usuarios">{{ usuario.nombre }}</li>
    </ul>
    <button (click)="actualizarLista()">Actualizar</button>
  `
})
export class ListaComponent {
  usuarios = [
    { id: 1, nombre: 'Ana' },
    { id: 2, nombre: 'Luis' },
    { id: 3, nombre: 'María' }
  ];

  actualizarLista(): void {
    // Recibe nueva respuesta del servidor (mismo contenido, nuevas referencias)
    this.usuarios = [
      { id: 1, nombre: 'Ana' },       // Mismo objeto, pero nueva referencia
      { id: 2, nombre: 'Luis' },      // Angular no sabe que es "el mismo"
      { id: 3, nombre: 'María García' } // Este SÍ cambió
    ];
    // Sin trackBy: destruye y recrea los 3 <li>
    // Con trackBy: solo recrea el <li> del id:3
  }
}
```

## La solución: trackBy

```typescript
@Component({
  standalone: true,
  imports: [CommonModule],
  template: `
    <!-- ✅ Con trackBy: Angular reutiliza nodos DOM cuando el ID no cambia -->
    <ul>
      <li *ngFor="let usuario of usuarios; trackBy: trackPorId">
        {{ usuario.nombre }}
      </li>
    </ul>
  `
})
export class ListaOptimizadaComponent {
  usuarios: Usuario[] = [];

  // La función recibe el índice y el elemento, debe devolver un identificador único
  trackPorId(index: number, usuario: Usuario): number {
    return usuario.id;
  }
}
```

## Cuándo es más importante trackBy

- **Listas largas**: cuanto más larga la lista, mayor el impacto.
- **Actualizaciones frecuentes**: si la lista se actualiza con peticiones HTTP periódicas.
- **Componentes complejos en cada fila**: destruir y recrear componentes con `ngOnInit`, suscripciones y lógica compleja es costoso.
- **Animaciones en las filas**: sin trackBy, las animaciones de entrada se disparan para todos los elementos en cada actualización.

## trackBy con valores primitivos

```typescript
// Para listas de strings o números
trackPorValor(index: number, valor: string): string {
  return valor;
}

// Para objetos sin ID (último recurso)
trackPorIndice(index: number): number {
  return index; // ⚠️ Evitar si el orden puede cambiar
}
```

## Nueva sintaxis @for (Angular 17+): track obligatorio

El nuevo bloque `@for` hace que `track` sea **obligatorio**, incentivando siempre la buena práctica:

```html
<!-- ✅ track es requerido en @for — no se puede omitir -->
@for (usuario of usuarios; track usuario.id) {
  <li>{{ usuario.nombre }}</li>
}

<!-- También válido: track por índice (menos óptimo) -->
@for (item of items; track $index) {
  <li>{{ item }}</li>
}
```

## Comparación de rendimiento

```
Sin trackBy (lista de 1000 elementos, 10 cambian):
  → Destruye 1000 nodos DOM
  → Crea 1000 nodos DOM nuevos
  → Dispara ngOnInit en 1000 componentes
  → Total: ~2000 operaciones DOM

Con trackBy (lista de 1000 elementos, 10 cambian):
  → Reutiliza 990 nodos DOM
  → Actualiza 10 nodos DOM
  → Dispara ngOnChanges en 10 componentes
  → Total: ~10 operaciones DOM significativas ✅
```

**Consejo de entrevista**: Siempre usa `trackBy` en listas que pueden cambiar. La función `trackBy` debe devolver un valor **único y estable**: el `id` de la entidad es la opción ideal. Evita usar el índice del array (`$index`) cuando los elementos pueden reordenarse o eliminarse, ya que en ese caso Angular podría reutilizar el nodo DOM equivocado. En Angular 17+, el nuevo `@for` obliga a definir `track`, lo que convierte esta buena práctica en un estándar del lenguaje de plantillas.
