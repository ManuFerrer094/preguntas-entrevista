---
title: ¿Cuáles son las diferencias entre los distintos proveedores de inyección de dependencias en Angular?
difficulty: hard
tags: [angular, dependency-injection, providers, injector]
---

El sistema de **inyección de dependencias (DI)** de Angular es jerárquico y altamente configurable. La forma en que se registra un proveedor determina cuántas instancias se crean, cuándo se destruyen y qué partes de la aplicación pueden acceder a ellas.

## Árbol de inyectores en Angular

```
PlatformInjector (bootstrap del módulo de plataforma)
    └── AppInjector / EnvironmentInjector (raíz de la aplicación)
            └── ElementInjector (árbol de componentes)
                    └── Por cada componente
```

## `providedIn: 'root'` — Singleton global (más común)

```typescript
@Injectable({
  providedIn: 'root'
})
export class AuthService {
  // Una sola instancia para toda la aplicación
}
```

- ✅ **Tree-shakeable**: si no se usa, no se incluye en el bundle.
- ✅ **Singleton**: la misma instancia en toda la app.
- ✅ El patrón **más recomendado** para servicios de aplicación.

## `providedIn: 'platform'` — Singleton de plataforma

```typescript
@Injectable({
  providedIn: 'platform'
})
export class SincronizacionService {}
```

- Crea una instancia compartida entre **múltiples aplicaciones Angular** en la misma página.
- Útil en micro-frontends donde hay varias apps Angular simultáneas.
- Raro en aplicaciones estándar.

## `providedIn: 'any'` — Una instancia por módulo lazy

```typescript
@Injectable({
  providedIn: 'any'
})
export class LogService {}
```

- Crea una instancia en el **inyector raíz** para módulos eager.
- Crea una **instancia separada** para cada módulo cargado con lazy loading.
- Útil cuando cada feature module necesita su propia instancia del servicio.

## Providers en `@NgModule`

```typescript
@NgModule({
  providers: [
    MiServicio,  // Forma corta

    // Forma completa con useClass
    { provide: MiServicio, useClass: MiServicioAlternativo },

    // useValue: para valores estáticos o de configuración
    { provide: API_URL, useValue: 'https://api.ejemplo.com' },

    // useFactory: creación dinámica
    {
      provide: MiServicio,
      useFactory: (config: ConfigService) => new MiServicio(config),
      deps: [ConfigService]
    },

    // useExisting: alias (apunta a otro token ya registrado)
    { provide: Logger, useExisting: ConsolaLogger }
  ]
})
export class FeatureModule {}
```

- **No tree-shakeable**: siempre se incluye aunque no se use.
- Crea una instancia por módulo (aunque se repite la instancia para módulos cargados eagerly en la misma jerarquía).

## Providers en `@Component` — Scope de componente

```typescript
@Component({
  selector: 'app-formulario',
  standalone: true,
  providers: [FormularioService]  // Nueva instancia por cada instancia del componente
})
export class FormularioComponent {
  constructor(private formService: FormularioService) {}
  // Esta instancia es DIFERENTE a la de otro FormularioComponent en la misma página
}
```

- Crea una instancia **separada para cada instancia** del componente en el DOM.
- Se destruye cuando el componente se destruye.
- Útil para servicios con estado local al componente.

## Tipos de providers (useClass, useValue, useFactory, useExisting)

### `useClass` — Sustituir implementación

```typescript
// En desarrollo/testing: usar implementación mock
providers: [
  {
    provide: UsuarioService,
    useClass: isDevMode() ? MockUsuarioService : UsuarioService
  }
]
```

### `useValue` — Valores estáticos y tokens

```typescript
// InjectionToken: para valores no-clase (strings, números, objetos)
export const API_CONFIG = new InjectionToken<ApiConfig>('ApiConfig', {
  providedIn: 'root',
  factory: () => ({
    baseUrl: 'https://api.ejemplo.com',
    timeout: 5000,
    version: 'v2'
  })
});

// Proveedor con token
providers: [
  { provide: API_CONFIG, useValue: { baseUrl: '/api', timeout: 3000, version: 'v1' } }
]

// Inyección del token
@Injectable({ providedIn: 'root' })
export class HttpService {
  constructor(@Inject(API_CONFIG) private config: ApiConfig) {
    console.log('API URL:', this.config.baseUrl);
  }
}
```

### `useFactory` — Creación dinámica

```typescript
providers: [
  {
    provide: LoggerService,
    useFactory: (configService: ConfigService, http: HttpClient) => {
      const config = configService.getConfig();
      return config.debug
        ? new VerboseLoggerService(http)
        : new SilentLoggerService();
    },
    deps: [ConfigService, HttpClient]  // Dependencias de la factory
  }
]
```

### `useExisting` — Alias entre tokens

```typescript
providers: [
  ConsolaLogger,  // Registrar la implementación real
  { provide: Logger, useExisting: ConsolaLogger }  // Logger apunta a ConsolaLogger
  // ✅ No crea una segunda instancia, reutiliza la existente (a diferencia de useClass)
]
```

## InjectionToken — Para valores no-clase

```typescript
// Definición del token
export const IDIOMA_TOKEN = new InjectionToken<string>('idioma.predeterminado', {
  factory: () => navigator.language ?? 'es'
});

export const ENTORNO_TOKEN = new InjectionToken<Entorno>('entorno', {
  providedIn: 'root',
  factory: () => ({
    produccion: false,
    apiUrl: 'https://dev-api.ejemplo.com'
  })
});

// Uso con inject() (Angular 14+)
@Injectable({ providedIn: 'root' })
export class LocalizacionService {
  private idioma = inject(IDIOMA_TOKEN);
  private entorno = inject(ENTORNO_TOKEN);
}
```

## Resumen: jerarquía y alcance

| Configuración              | Instancias          | Tree-Shakeable | Destrucción                    |
|----------------------------|---------------------|----------------|--------------------------------|
| `providedIn: 'root'`       | 1 (global)          | ✅ Sí          | Al cerrar la app               |
| `providedIn: 'platform'`   | 1 (plataforma)      | ✅ Sí          | Al cerrar la plataforma        |
| `providedIn: 'any'`        | 1 por módulo lazy   | ✅ Sí          | Al destruir el módulo          |
| `@NgModule.providers`      | 1 por módulo        | ❌ No          | Al destruir el módulo          |
| `@Component.providers`     | 1 por componente    | ❌ No          | Al destruir el componente      |

**Consejo de entrevista**: La pregunta clave es: "¿cuántas instancias se crean y cuándo se destruyen?" La respuesta depende del nivel del inyector. Un error común es registrar un servicio en los `providers` de un `SharedModule` y esperar que sea singleton: como el SharedModule puede importarse en múltiples módulos, se crean múltiples instancias. La solución correcta es `providedIn: 'root'`. Otro tema avanzado es `InjectionToken`, que es la forma correcta de inyectar valores primitivos (strings, numbers, configuración) sin necesidad de clases envolventes.
