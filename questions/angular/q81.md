Los **micro-frontends** son una arquitectura donde una aplicación web frontend se divide en múltiples aplicaciones independientes más pequeñas, desarrolladas y desplegadas de forma autónoma por distintos equipos. Es la extensión del concepto de microservicios al frontend.

## Conceptos fundamentales

```
┌─────────────────────────────────────┐
│         Shell Application          │
│  (Host: enrutamiento global)        │
├────────────┬────────────┬───────────┤
│   MFE 1    │   MFE 2    │   MFE 3   │
│  (Ventas)  │ (Inventario)│ (Reportes)│
│  Equipo A  │  Equipo B  │  Equipo C  │
└────────────┴────────────┴───────────┘
```

Cada micro-frontend:
- Tiene su propio repositorio y pipeline de CI/CD
- Se despliega independientemente
- Se puede actualizar sin redeployar el Shell

## Webpack Module Federation con Angular

La solución más común para micro-frontends en Angular es **Module Federation** (Webpack 5).

### Configuración del Shell (Host)

```typescript
// webpack.config.js del Shell
const ModuleFederationPlugin = require('webpack/lib/container/ModuleFederationPlugin');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'shell',
      remotes: {
        // Declarar las aplicaciones remotas
        ventas: 'ventas@http://localhost:4201/remoteEntry.js',
        inventario: 'inventario@http://localhost:4202/remoteEntry.js',
      },
      shared: {
        '@angular/core': { singleton: true, strictVersion: true },
        '@angular/common': { singleton: true, strictVersion: true },
        '@angular/router': { singleton: true, strictVersion: true },
      }
    })
  ]
};

// app.routes.ts del Shell
const routes: Routes = [
  {
    path: 'ventas',
    loadChildren: () =>
      loadRemoteModule({
        type: 'module',
        remoteEntry: 'http://localhost:4201/remoteEntry.js',
        exposedModule: './Module'
      }).then(m => m.VentasModule)
  }
];
```

### Configuración del Micro-Frontend (Remote)

```typescript
// webpack.config.js del MFE de Ventas
const ModuleFederationPlugin = require('webpack/lib/container/ModuleFederationPlugin');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'ventas',
      filename: 'remoteEntry.js',
      exposes: {
        // Exponer módulos/componentes al Shell
        './Module': './src/app/ventas/ventas.module.ts',
        './Component': './src/app/ventas/ventas.component.ts',
      },
      shared: {
        '@angular/core': { singleton: true, strictVersion: true },
        '@angular/common': { singleton: true, strictVersion: true },
      }
    })
  ]
};
```

### Con @angular-architects/module-federation (recomendado)

```bash
# Instalar el plugin
ng add @angular-architects/module-federation --project shell --type host --port 4200
ng add @angular-architects/module-federation --project ventas --type remote --port 4201
```

```typescript
// module-federation.config.ts del Shell (generado automáticamente)
export const mfConfig: ManifestFile = {
  ventas: 'http://localhost:4201/remoteEntry.js',
  inventario: 'http://localhost:4202/remoteEntry.js',
};

// shell/app.routes.ts
import { loadRemoteModule } from '@angular-architects/module-federation';

const routes: Routes = [
  {
    path: 'ventas',
    loadComponent: () =>
      loadRemoteModule('ventas', './Component')
        .then(m => m.VentasRootComponent)
  }
];
```

## Comunicación entre micro-frontends

```typescript
// shared-events.service.ts — Servicio compartido para comunicación
// (Se comparte como singleton gracias a Module Federation)
@Injectable({ providedIn: 'root' })
export class SharedEventsService {
  private eventBus = new Subject<{ tipo: string; payload: unknown }>();

  eventos$ = this.eventBus.asObservable();

  emitir(tipo: string, payload: unknown) {
    this.eventBus.next({ tipo, payload });
  }
}

// MFE de Ventas emite un evento
this.sharedEvents.emitir('PEDIDO_CREADO', { pedidoId: 123 });

// MFE de Inventario escucha el evento
this.sharedEvents.eventos$.pipe(
  filter(e => e.tipo === 'PEDIDO_CREADO')
).subscribe(e => this.actualizarStock(e.payload));
```

## Ventajas e inconvenientes

| Aspecto          | Ventaja                                   | Inconveniente                         |
|------------------|-------------------------------------------|---------------------------------------|
| Equipos          | Autonomía total por equipo                | Coordinación de versiones compartidas |
| Despliegue       | Deploy independiente por feature          | Complejidad de infraestructura        |
| Tecnología       | Posible usar distintos frameworks         | Bundle duplicado si no se comparte    |
| Escalabilidad    | Equipos grandes sin bloqueos              | Debugging más complejo                |
| Rendimiento      | Carga solo lo necesario                  | Overhead de Module Federation         |

**Consejo de entrevista**: Los micro-frontends resuelven problemas organizacionales (equipos grandes, dominios separados) más que técnicos. No son recomendables para equipos pequeños o aplicaciones pequeñas. La clave de Module Federation es `singleton: true` para librerías compartidas (Angular, RxJS) —sin esto, se cargarían múltiples versiones en la misma página, lo que causa errores difíciles de depurar.
