**Angular DevTools** es una extensión del navegador (Chrome y Firefox) desarrollada por el equipo de Angular para **inspeccionar y depurar** aplicaciones Angular directamente desde las herramientas de desarrollo. Permite explorar el árbol de componentes, inspeccionar el estado, y analizar el rendimiento del change detection.

## Instalación

1. Instalar la extensión **Angular DevTools** desde Chrome Web Store o Firefox Add-ons.
2. Abrir DevTools del navegador (F12).
3. Pestaña **Angular** disponible solo cuando la app tiene modo de desarrollo activo.

> La extensión no funciona en producción (build con `--configuration=production`) a menos que Angular lo permita explícitamente.

## Inspector de componentes

```
Panel: Components
├── AppComponent
│   ├── NavbarComponent
│   ├── RouterOutlet
│   │   └── DashboardComponent
│   │       ├── MetricsCardComponent (x3)
│   │       └── ChartComponent
│   └── FooterComponent
```

Al seleccionar un componente puedes:
- Ver sus **@Input()** y su valor actual
- Ver sus **@Output()**
- Ver sus **signals** y valores computados
- Ver el estado del **FormGroup** (si usa formularios reactivos)
- Editar valores en tiempo real

## Profiler de Change Detection

```typescript
// El Profiler registra qué componentes se re-renderizaron y por qué

// Ejemplo: componente con OnPush optimizado
@Component({
  selector: 'app-card',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<h3>{{ titulo }}</h3>`
})
export class CardComponent {
  @Input() titulo = '';
}

// En el Profiler verás:
// - CardComponent: NO re-renderizado (OnPush + input no cambió)
// - vs componentes Default: re-renderizan en cada ciclo
```

### Cómo usar el Profiler

1. Ir a la pestaña **Profiler** en Angular DevTools.
2. Hacer clic en **Record** (círculo rojo).
3. Interactuar con la aplicación.
4. Hacer clic en **Stop**.
5. Analizar el diagrama de llama (*flame graph*) que muestra:
   - Cuánto tiempo tardó cada ciclo de change detection
   - Qué componentes se verificaron
   - Cuáles causaron más overhead

## Depurar con ng en la consola

Angular DevTools también expone la función `ng` en la consola del navegador:

```javascript
// Seleccionar un elemento en el inspector y luego en consola:

// Obtener la instancia del componente seleccionado
const comp = ng.getComponent($0);
comp.titulo = 'Nuevo título';     // Cambiar una propiedad
ng.applyChanges(comp);            // Forzar change detection

// Obtener el contexto del contexto de la directiva
ng.getDirectives($0);

// Ver los ownerContext (señales/computed values)
ng.getOwningComponent($0);

// Obtener el host element de un componente
ng.getHostElement(componentInstance);
```

## Depurar Signals con Angular DevTools

```typescript
@Component({
  standalone: true,
  template: `
    <p>Contador: {{ contador() }}</p>
    <p>Doble: {{ doble() }}</p>
  `
})
export class SignalComponent {
  contador = signal(0);
  doble = computed(() => this.contador() * 2);
  // En Angular DevTools podrás ver el valor actual de cada signal
  // y cuándo se recalculan los computed signals
}
```

## Tips de depuración avanzada

```typescript
// Habilitar debugging en producción (temporal, no hacer en deploy real)
// main.ts
import { enableDebugTools } from '@angular/platform-browser';

bootstrapApplication(AppComponent).then(ref => {
  if (!environment.production) {
    enableDebugTools(ref.components[0]);
  }
});
```

```typescript
// Usar console.log con signals para rastrear cambios
effect(() => {
  console.group('🔄 Change Detection Cycle');
  console.log('contador:', this.contador());
  console.log('filtro:', this.filtro());
  console.groupEnd();
});
```

## Qué detectar con DevTools en una entrevista técnica

| Problema                           | Herramienta DevTools              |
|------------------------------------|-----------------------------------|
| Componente re-renderiza demasiado  | Profiler → ver tiempo y frecuencia |
| @Input no llega al hijo            | Components → inspect inputs        |
| Signal no actualiza la vista       | Components → ver signal values     |
| Change detection lento             | Profiler → flame graph             |
| Memoria no se libera               | Components → verificar destrucción |

**Consejo de entrevista**: Mencionar Angular DevTools demuestra madurez en el desarrollo Angular. El Profiler es especialmente valioso para identificar componentes que se re-renderizan innecesariamente y validar que `ChangeDetectionStrategy.OnPush` está funcionando correctamente. En una entrevista técnica, describir cómo usarías el Profiler para optimizar una aplicación lenta es una respuesta de nivel senior.
