La **hidratación** (Hydration) es el proceso de convertir HTML renderizado en el servidor (SSR) en una aplicación Angular completamente interactiva en el cliente, **sin destruir y recrear el DOM existente**.

## El problema sin hidratación

Cuando Angular usa SSR sin hidratación, el proceso es:
1. El servidor genera HTML estático y lo envía al cliente.
2. El cliente lo muestra inmediatamente (First Contentful Paint rápido).
3. Angular carga en el cliente y **destruye todo el DOM** que generó el servidor.
4. Angular **recrea todo el DOM** desde cero.

Esto causa un **parpadeo visual** (flicker) y desperdicia el trabajo del servidor.

## Con hidratación (Angular 17+)

```
Servidor:   genera HTML → lo envía al cliente
Cliente:    muestra el HTML del servidor (rápido) ✅
Angular:    "hidratar" el DOM existente → añade event listeners → app interactiva ✅
            NO destruye ni recrea el DOM → sin parpadeo ✅
```

## Configuración de la hidratación

```typescript
// app.config.ts (Angular 17+)
import { ApplicationConfig } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideClientHydration, withEventReplay } from '@angular/platform-browser';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideClientHydration(
      withEventReplay()  // Captura eventos antes de la hidratación (Angular 18+)
    )
  ]
};
```

```typescript
// main.server.ts (servidor)
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { config } from './app/app.config.server';

const bootstrap = () => bootstrapApplication(AppComponent, config);
export default bootstrap;
```

## withEventReplay()

Una de las mejoras más importantes. Sin `withEventReplay()`, si el usuario hace clic en un botón **antes de que Angular termine de hidratarse**, el evento se pierde.

Con `withEventReplay()`, Angular captura los eventos del usuario durante la hidratación y los **reproduce** una vez que la app está lista.

```typescript
provideClientHydration(
  withEventReplay(),         // Captura y reproduce eventos
  withIncrementalHydration() // Angular 19+: hidrata solo las partes visibles
)
```

## Hidratación incremental (Angular 19+)

Angular 19 introduce la hidratación incremental con la directiva `@defer`:

```html
<!-- Solo se hidratan cuando el usuario interactúa o cuando son visibles -->
@defer (hydrate on viewport) {
  <app-comentarios></app-comentarios>
}

@defer (hydrate on interaction) {
  <app-galeria-imagenes></app-galeria-imagenes>
}

@defer (hydrate on idle) {
  <app-widget-recomendaciones></app-widget-recomendaciones>
}
```

## Beneficios en Core Web Vitals

La hidratación mejora directamente las métricas de rendimiento de Google:

| Métrica | Sin hidratación | Con hidratación |
|---------|-----------------|-----------------|
| LCP (Largest Contentful Paint) | Lento (rebuild) | Rápido (reutiliza HTML) |
| CLS (Cumulative Layout Shift)  | Alto (parpadeo) | Bajo (sin recreación)  |
| FID / INP (Interactivity)      | Retrasado       | Mejorado con withEventReplay |
| TTI (Time to Interactive)      | Mayor           | Menor                         |

## Restricciones de la hidratación

Para que la hidratación funcione correctamente, hay algunas restricciones:
- El HTML generado en el servidor y en el cliente debe ser **idéntico**.
- No usar `document.write()` ni manipulaciones directas del DOM antes de la hidratación.
- Evitar acceder a `localStorage` o `window` durante el renderizado del servidor.

```typescript
// ✅ Patrón seguro para código que solo debe ejecutarse en el cliente
import { isPlatformBrowser } from '@angular/common';
import { PLATFORM_ID, inject } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class StorageService {
  private platformId = inject(PLATFORM_ID);

  getItem(key: string): string | null {
    if (isPlatformBrowser(this.platformId)) {
      return localStorage.getItem(key); // Solo en el cliente
    }
    return null;
  }
}
```

## Configurar SSR con hidratación

```bash
# Crear nuevo proyecto con SSR y hidratación
ng new mi-app --ssr

# Añadir SSR a un proyecto existente
ng add @angular/ssr
```

**Consejo de entrevista**: La hidratación fue introducida en Angular 16 (developer preview) y se volvió estable en Angular 17. Es la mejora de rendimiento más significativa para aplicaciones con SSR, ya que elimina el parpadeo (flicker) y reduce el tiempo hasta interactividad. `withEventReplay()` es la respuesta al problema de "eventos perdidos durante la hidratación". En Angular 19+, la hidratación incremental con `@defer` lleva esto al siguiente nivel, hidratando solo las partes que se necesitan cuando se necesitan.
