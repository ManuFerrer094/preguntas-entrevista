---
title: ¿Cuáles son los lifecycle hooks disponibles en Angular?
difficulty: medium
tags: [angular, lifecycle, hooks]
---

Los **lifecycle hooks** (ganchos del ciclo de vida) son métodos especiales que Angular llama automáticamente en momentos clave del ciclo de vida de un componente o directiva. Permiten ejecutar lógica personalizada en respuesta a eventos como la creación, actualización y destrucción del componente.

## Orden de ejecución

```
constructor()
    ↓
ngOnChanges()    ← se ejecuta antes de ngOnInit si hay @Input()
    ↓
ngOnInit()
    ↓
ngDoCheck()
    ↓
ngAfterContentInit()
    ↓
ngAfterContentChecked()
    ↓
ngAfterViewInit()
    ↓
ngAfterViewChecked()
    ↓
ngOnDestroy()
```

## Descripción de cada hook

### `ngOnChanges(changes: SimpleChanges)`
Se ejecuta **antes de `ngOnInit`** y cada vez que cambia un `@Input()`. Recibe un objeto `SimpleChanges` con los valores anteriores y actuales.

```typescript
ngOnChanges(changes: SimpleChanges): void {
  if (changes['userId']) {
    console.log('Anterior:', changes['userId'].previousValue);
    console.log('Actual:', changes['userId'].currentValue);
  }
}
```

### `ngOnInit()`
Se ejecuta **una sola vez** tras la primera ejecución de `ngOnChanges`. Es el lugar ideal para la lógica de inicialización, peticiones HTTP y configuración inicial.

```typescript
ngOnInit(): void {
  this.userService.getUser(this.userId).subscribe(user => {
    this.user = user;
  });
}
```

### `ngDoCheck()`
Se ejecuta en **cada ciclo de detección de cambios**. Permite implementar lógica de detección personalizada. Úsalo con precaución ya que se invoca con mucha frecuencia.

### `ngAfterContentInit()`
Se ejecuta **una sola vez** después de que Angular proyecta contenido externo en el componente (contenido dentro de `<ng-content>`). Útil para acceder a `@ContentChild` / `@ContentChildren`.

```typescript
@ContentChild(TabComponent) tabContent!: TabComponent;

ngAfterContentInit(): void {
  console.log('Contenido proyectado listo:', this.tabContent);
}
```

### `ngAfterContentChecked()`
Se ejecuta después de cada verificación del contenido proyectado. Se llama tras `ngAfterContentInit` y cada `ngDoCheck`.

### `ngAfterViewInit()`
Se ejecuta **una sola vez** después de que Angular inicializa la vista del componente y sus vistas hijas. Ideal para acceder a `@ViewChild` / `@ViewChildren` y manipulaciones del DOM.

```typescript
@ViewChild('miInput') miInput!: ElementRef;

ngAfterViewInit(): void {
  this.miInput.nativeElement.focus();
}
```

### `ngAfterViewChecked()`
Se ejecuta después de cada verificación de la vista del componente y sus vistas hijas.

### `ngOnDestroy()`
Se ejecuta **justo antes** de que Angular destruya el componente. Es **crítico** para limpiar recursos y evitar memory leaks.

```typescript
private destroy$ = new Subject<void>();

ngOnInit(): void {
  this.dataService.getData()
    .pipe(takeUntil(this.destroy$))
    .subscribe(data => this.data = data);
}

ngOnDestroy(): void {
  this.destroy$.next();
  this.destroy$.complete();
}
```

## Hooks más importantes en entrevistas

| Hook              | Frecuencia  | Uso principal                                    |
|-------------------|-------------|--------------------------------------------------|
| `ngOnInit`        | Una vez     | Inicialización, llamadas HTTP                    |
| `ngOnChanges`     | Cada cambio | Reaccionar a cambios de `@Input()`               |
| `ngOnDestroy`     | Una vez     | Cancelar suscripciones, limpiar timers           |
| `ngAfterViewInit` | Una vez     | Acceder a elementos del DOM con `@ViewChild`     |

## Implementación de interfaces (recomendado)

```typescript
import { Component, OnInit, OnDestroy, OnChanges, SimpleChanges } from '@angular/core';

@Component({ selector: 'app-ejemplo', template: '' })
export class EjemploComponent implements OnInit, OnDestroy, OnChanges {
  ngOnChanges(changes: SimpleChanges): void { /* ... */ }
  ngOnInit(): void { /* ... */ }
  ngOnDestroy(): void { /* ... */ }
}
```

**Consejo de entrevista**: Los tres hooks más preguntados son `ngOnInit`, `ngOnDestroy` y `ngOnChanges`. Recuerda que `ngOnChanges` solo detecta cambios de **referencia** en los `@Input()`, no mutaciones internas de objetos o arrays. Para detectar cambios profundos, deberías usar `ngDoCheck` o `ChangeDetectorRef`.
