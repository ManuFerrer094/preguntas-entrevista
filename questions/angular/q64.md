---
title: ¿Cuáles son las formas de comunicación entre componentes en Angular?

En Angular existen múltiples formas de comunicar componentes, cada una apropiada para diferentes relaciones (padre-hijo, hermanos, no relacionados) y diferentes escalas de complejidad.

## 1. Padre → Hijo: `@Input()`

La forma más simple y directa. El padre pasa datos al hijo mediante property binding.

```typescript
// hijo.component.ts
@Component({
  selector: 'app-tarjeta',
  standalone: true,
  template: `<h3>{{ titulo }}</h3><p>{{ descripcion }}</p>`
})
export class TarjetaComponent {
  @Input() titulo!: string;
  @Input() descripcion = '';
}
```

```html
<!-- padre.component.html -->
<app-tarjeta [titulo]="producto.nombre" [descripcion]="producto.descripcion">
</app-tarjeta>
```

## 2. Hijo → Padre: `@Output()` con `EventEmitter`

```typescript
@Component({
  selector: 'app-formulario-busqueda',
  standalone: true,
  imports: [FormsModule],
  template: `
    <input [(ngModel)]="termino">
    <button (click)="buscar()">🔍</button>
  `
})
export class FormularioBusquedaComponent {
  termino = '';
  @Output() busquedaRealizada = new EventEmitter<string>();

  buscar(): void {
    this.busquedaRealizada.emit(this.termino);
  }
}
```

```html
<app-formulario-busqueda (busquedaRealizada)="onBusqueda($event)">
</app-formulario-busqueda>
```

## 3. Padre → Hijo: `@ViewChild()`

El padre accede directamente a la instancia o al DOM del hijo:

```typescript
@Component({
  selector: 'app-padre',
  standalone: true,
  imports: [HijoComponent],
  template: `
    <app-hijo #hijo></app-hijo>
    <button (click)="llamarMetodoHijo()">Llamar método</button>
  `
})
export class PadreComponent {
  @ViewChild('hijo') hijo!: HijoComponent;
  @ViewChild(HijoComponent) hijoRef!: HijoComponent; // Por tipo

  llamarMetodoHijo(): void {
    this.hijo.reiniciar();     // Llamar método del hijo
    console.log(this.hijo.valor); // Leer estado del hijo
  }
}
```

## 4. Entre componentes no relacionados: Servicio compartido

Para componentes que no tienen relación padre-hijo, la solución más común es un servicio con `BehaviorSubject` o Signals:

```typescript
// carrito.service.ts
@Injectable({ providedIn: 'root' })
export class CarritoService {
  private items$ = new BehaviorSubject<Producto[]>([]);

  readonly items = this.items$.asObservable();
  readonly cantidad = this.items$.pipe(map(items => items.length));

  agregar(producto: Producto): void {
    this.items$.next([...this.items$.value, producto]);
  }

  vaciar(): void {
    this.items$.next([]);
  }
}
```

```typescript
// producto.component.ts — componente A
export class ProductoComponent {
  constructor(private carrito: CarritoService) {}
  agregarAlCarrito(): void { this.carrito.agregar(this.producto); }
}

// navbar.component.ts — componente B (no relacionado con A)
export class NavbarComponent {
  cantidad$ = this.carrito.cantidad;
  constructor(private carrito: CarritoService) {}
}
```

## 5. Con Signals (Angular 16+)

Signals son la forma moderna y reactiva de compartir estado:

```typescript
// contador.service.ts
@Injectable({ providedIn: 'root' })
export class ContadorService {
  private _contador = signal(0);

  readonly contador = this._contador.asReadonly();
  readonly esPositivo = computed(() => this._contador() > 0);

  incrementar(): void { this._contador.update(n => n + 1); }
  decrementar(): void { this._contador.update(n => n - 1); }
  reiniciar(): void { this._contador.set(0); }
}
```

```html
<!-- Cualquier componente puede leer el signal directamente -->
<p>Contador: {{ contadorService.contador() }}</p>
```

## 6. A través de la ruta: parámetros y query params

```typescript
// Componente A — navegar con parámetros
this.router.navigate(['/detalle', id], {
  queryParams: { modo: 'edicion' }
});

// Componente B — recibir parámetros
constructor(private route: ActivatedRoute) {}
ngOnInit(): void {
  const id = this.route.snapshot.params['id'];
  const modo = this.route.snapshot.queryParams['modo'];
}
```

## 7. Gestión de estado global: NgRx

Para aplicaciones complejas con estado compartido a gran escala:

```typescript
// Con NgRx Store
@Component({ /* ... */ })
export class ProductosComponent {
  productos$ = this.store.select(selectTodosLosProductos);
  constructor(private store: Store) {}

  cargar(): void {
    this.store.dispatch(ProductosActions.cargar());
  }
}
```

## Resumen: cuándo usar cada método

| Relación                          | Solución recomendada                          |
|-----------------------------------|-----------------------------------------------|
| Padre → Hijo (datos simples)      | `@Input()`                                    |
| Hijo → Padre (eventos)            | `@Output()` + `EventEmitter`                  |
| Padre accede a hijo               | `@ViewChild()`                                |
| Padre accede a contenido proyectado | `@ContentChild()`                           |
| Componentes hermanos/no relacionados | Servicio con `BehaviorSubject` o Signals   |
| Estado complejo de la app         | NgRx Store / Signal Store                     |
| Datos entre rutas                 | Route params / Query params / Route data      |

**Consejo de entrevista**: La evolución en Angular es: `@Input/@Output` para relación directa padre-hijo, Servicios con `BehaviorSubject` para estado compartido simple, y NgRx para estado complejo. Con Angular 16+, los **Signals** están reemplazando gradualmente a `BehaviorSubject` para el estado en servicios, ya que son más simples, sincrónicos y tienen mejor integración con el sistema de change detection.
