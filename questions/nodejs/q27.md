---
title: ¿Qué es PM2 y cómo gestionar procesos Node.js en producción?
difficulty: medium
tags: [nodejs, pm2, production, process-manager, deployment]
---

**PM2** es un gestor de procesos para Node.js en producción. Mantiene aplicaciones en ejecución continua, gestiona múltiples instancias para aprovechar todos los núcleos del CPU, proporciona logging, monitorización y reinicio automático ante fallos.

## Instalación y comandos básicos

```bash
npm install -g pm2

# Iniciar una aplicación
pm2 start app.js

# Iniciar con nombre
pm2 start app.js --name "mi-api"

# Modo cluster: utilizar todos los núcleos del CPU
pm2 start app.js --name "mi-api" -i max
# -i max: usa todos los núcleos
# -i 4: usa exactamente 4 instancias
# -i 0: usa todos los CPUs disponibles

# Ver aplicaciones en ejecución
pm2 list

# Ver logs en tiempo real
pm2 logs mi-api
pm2 logs --lines 100  # Últimas 100 líneas

# Monitorización interactiva
pm2 monit

# Reiniciar / Detener / Eliminar
pm2 restart mi-api
pm2 reload mi-api     # Zero-downtime reload (cluster mode)
pm2 stop mi-api
pm2 delete mi-api
```

## Archivo de configuración ecosystem.config.js

```javascript
// ecosystem.config.js — Configuración declarativa de PM2
module.exports = {
  apps: [
    {
      // API principal
      name: 'api-produccion',
      script: './src/server.js',

      // Cluster mode: múltiples instancias
      instances: 'max',         // O un número específico
      exec_mode: 'cluster',     // 'fork' para una sola instancia

      // Reinicio automático
      watch: false,             // No vigilar archivos en producción
      max_memory_restart: '500M',  // Reiniciar si supera 500MB de RAM

      // Variables de entorno
      env: {
        NODE_ENV: 'development',
        PORT: 3000
      },
      env_production: {
        NODE_ENV: 'production',
        PORT: 80
      },

      // Logs
      log_file: './logs/combined.log',
      out_file: './logs/out.log',
      error_file: './logs/error.log',
      log_date_format: 'YYYY-MM-DD HH:mm:ss Z',
      merge_logs: true,          // Unificar logs de todas las instancias

      // Reinicio en ciertas condiciones
      min_uptime: '5s',         // Mínimo tiempo de vida para considerarse "estable"
      max_restarts: 10,         // Máximo de reinicios antes de dejar de reintentar

      // Graceful shutdown
      kill_timeout: 5000,       // Tiempo para terminar limpiamente
      listen_timeout: 10000,    // Tiempo para que el proceso esté listo
    },

    {
      // Worker para jobs en background
      name: 'worker-jobs',
      script: './src/workers/jobs.js',
      instances: 1,              // Solo una instancia (evitar jobs duplicados)
      exec_mode: 'fork',
      env_production: {
        NODE_ENV: 'production',
      }
    }
  ]
};

// Comandos con el ecosystem:
// pm2 start ecosystem.config.js
// pm2 start ecosystem.config.js --env production
// pm2 reload ecosystem.config.js --env production
```

## Graceful shutdown en la aplicación

```javascript
// server.js — Manejar señales de PM2 para apagado limpio
import express from 'express';
import { createServer } from 'http';

const app = express();
const server = createServer(app);

app.get('/health', (req, res) => res.json({ status: 'ok' }));

server.listen(3000, () => {
  console.log('Servidor iniciado en puerto 3000');
  // Notificar a PM2 que el proceso está listo (cluster mode)
  if (process.send) process.send('ready');
});

// Graceful shutdown: terminar conexiones activas antes de cerrar
process.on('SIGINT', gracefulShutdown);
process.on('SIGTERM', gracefulShutdown);

async function gracefulShutdown(signal) {
  console.log(`[${signal}] Iniciando apagado limpio...`);

  // Dejar de aceptar nuevas conexiones
  server.close(async () => {
    console.log('Conexiones HTTP cerradas');

    try {
      // Cerrar conexiones a base de datos
      await db.disconnect();
      console.log('Base de datos desconectada');

      process.exit(0);
    } catch (error) {
      console.error('Error durante el apagado:', error);
      process.exit(1);
    }
  });

  // Forzar apagado si tarda demasiado
  setTimeout(() => {
    console.error('Forzando apagado...');
    process.exit(1);
  }, 10000);
}
```

## Monitorización y alertas

```bash
# Ver uso de CPU y memoria en tiempo real
pm2 monit

# Información detallada de un proceso
pm2 show mi-api

# Estadísticas generales
pm2 info mi-api

# Limpiar logs
pm2 flush

# Rotar logs (con pm2-logrotate)
pm2 install pm2-logrotate
pm2 set pm2-logrotate:max_size 100M
pm2 set pm2-logrotate:retain 30  # Mantener 30 días
```

## Inicio automático al arrancar el servidor

```bash
# Generar script de arranque automático para el sistema operativo
pm2 startup

# Ejecutar el comando que te dé como output (con sudo si es necesario):
# sudo env PATH=$PATH:/usr/bin pm2 startup systemd -u ubuntu --hp /home/ubuntu

# Guardar el estado actual de los procesos
pm2 save

# Ahora los procesos se reinician automáticamente si el servidor se reinicia
```

## Zero-downtime deployment

```bash
# Proceso de deploy sin interrupción del servicio:

# 1. Subir el nuevo código
git pull origin main

# 2. Instalar dependencias
npm ci --production

# 3. Reload con zero-downtime (solo cluster mode)
pm2 reload ecosystem.config.js --env production

# PM2 reinicia las instancias de una en una,
# manteniendo las demás en servicio durante el proceso
```

**Consejo de entrevista**: PM2 en modo cluster aprovecha todos los núcleos del CPU usando el módulo `cluster` de Node.js internamente. El flag `--update-env` en reload actualiza las variables de entorno sin reiniciar completamente. Para containerización (Docker/Kubernetes), PM2 no es necesario ya que el orquestador gestiona los reinicios —en ese caso, usar Node.js directamente con configuración de healthchecks. Menciona `pm2 reload` vs `pm2 restart`: el reload hace rolling update (zero-downtime), el restart detiene y reinicia todos los procesos a la vez.
