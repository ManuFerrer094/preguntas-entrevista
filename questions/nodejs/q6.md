---
title: ¿Qué es el módulo cluster en Node.js?
difficulty: hard

El módulo `cluster` de Node.js permite crear múltiples procesos hijos (workers) que comparten el mismo puerto del servidor. Ya que Node.js es single-threaded, `cluster` es la forma nativa de aprovechar todos los núcleos de CPU disponibles, escalando verticalmente la aplicación.

Por defecto, Node.js usa un solo hilo y un solo núcleo de CPU. Con `cluster`, puedes lanzar N procesos (típicamente uno por núcleo), cada uno con su propio event loop, que se distribuyen las conexiones entrantes.

```javascript
// cluster.js
const cluster = require('cluster');
const http = require('http');
const os = require('os');
const process = require('process');

const numCPUs = os.cpus().length;

if (cluster.isPrimary) {
  console.log(`Proceso primario ${process.pid} iniciado`);
  console.log(`Creando ${numCPUs} workers para ${numCPUs} CPUs...`);

  // Crear un worker por cada CPU
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  // Escuchar la muerte de workers y reemplazarlos
  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} murió (${signal || code}). Reiniciando...`);
    cluster.fork();
  });

  cluster.on('online', (worker) => {
    console.log(`Worker ${worker.process.pid} en línea`);
  });

} else {
  // Los workers comparten el servidor TCP
  http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Hola desde worker ${process.pid}\n`);
  }).listen(3000);

  console.log(`Worker ${process.pid} iniciado`);
}
```

## Comunicación entre procesos (IPC)

```javascript
// En el proceso primario
cluster.workers[workerId].send({ type: 'config', data: newConfig });

// En el worker
process.on('message', (msg) => {
  if (msg.type === 'config') {
    updateConfig(msg.data);
  }
});

// El worker también puede enviar mensajes al primario
process.send({ type: 'stats', requests: requestCount });
```

## Uso con Express

```javascript
const cluster = require('cluster');
const express = require('express');
const os = require('os');

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  for (let i = 0; i < numCPUs; i++) cluster.fork();

  cluster.on('exit', () => cluster.fork());
} else {
  const app = express();
  // ... configurar app
  app.listen(3000, () => {
    console.log(`Worker ${process.pid} escuchando en puerto 3000`);
  });
}
```

## cluster vs Worker Threads vs PM2

| | Cluster | Worker Threads | PM2 |
|---|---|---|---|
| Nivel | Proceso | Hilo | Proceso |
| Memoria compartida | No | Sí (SharedArrayBuffer) | No |
| Caso de uso | Servidores HTTP | CPU intensivo | Producción |
| Complejidad | Media | Alta | Baja |

**PM2 como alternativa más sencilla:**

```bash
# PM2 gestiona el cluster automáticamente
pm2 start app.js -i max  # -i max = un proceso por CPU
pm2 start app.js -i 4    # 4 procesos
```

**Consejo de entrevista**: En producción, se recomienda usar **PM2** en lugar del módulo `cluster` directamente, ya que PM2 gestiona automáticamente el clustering, los reinicios, los logs y el monitoreo. Sin embargo, entender `cluster` es importante para comprender cómo funciona la concurrencia en Node.js. Para cargas de trabajo CPU-intensivas (procesamiento de imágenes, cálculos matemáticos), `Worker Threads` es más apropiado que `cluster`.
