Si el puesto menciona Web Components de forma explícita, esta es una pregunta casi obligatoria. La clave es no recitar una lista, sino explicar qué resuelve cada pieza y cómo encajan juntas para construir componentes reutilizables nativos del navegador.

## Qué evalúa el entrevistador
- Si conoces los tres pilares que suelen pedirse en entrevista.
- Si distingues definición del componente, encapsulación y reutilización de marcado.
- Si puedes explicar por qué esto importa incluso cuando luego trabajas con una librería como Lit.

## Respuesta sólida
- `Custom Elements`: permiten definir nuevas etiquetas HTML con comportamiento propio, como `<bank-account-card>`.
- `Shadow DOM`: aporta encapsulación de estructura y estilos para que el CSS global no rompa el componente ni el componente contamine el resto.
- `HTML Templates`: permiten declarar marcado inerte que puedes clonar cuando lo necesitas.

## Cómo conectarlos
Un flujo típico sería:
- defines un `Custom Element`,
- renderizas dentro de su `Shadow DOM`,
- y reutilizas estructura a partir de un `template` o desde una librería que genere ese DOM.

## Matiz útil en entrevista
Web Components no significan "hacer todo a mano". Significan apoyarte en estándares del navegador. Luego puedes usar utilidades o librerías encima, pero el contrato base sigue siendo estándar y portable.

## Error común
Olvidar que `HTML Templates` forman parte de la historia. Mucha gente menciona solo `Custom Elements` y `Shadow DOM`, pero en entrevistas técnicas suele sumar explicar el papel de `<template>` como mecanismo de marcado reutilizable e inerte.

## Frase corta de entrevista
> Los pilares de Web Components son `Custom Elements` para definir la etiqueta, `Shadow DOM` para encapsularla y `HTML Templates` para reutilizar marcado sin ejecutarlo al parsear.
