Aunque la pregunta viva dentro de JavaScript, aquí están midiendo criterio frontend real. En SPAs grandes el problema no es escribir estilos, sino mantenerlos estables cuando crecen pantallas, estados, equipos y componentes.

## Qué evalúa el entrevistador
- Si sabes estructurar estilos por capas o por responsabilidades.
- Si entiendes cuándo SASS o LESS aportan valor y cuándo solo añaden complejidad.
- Si puedes convivir con utilidades o frameworks como Bootstrap sin perder control del sistema.

## Respuesta sólida
- Separaría tokens y variables, mixins/utilidades, estilos base y estilos por componente o feature.
- Evitaría selectores demasiado profundos, dependencias implícitas entre pantallas y sobreescrituras constantes de estilos globales.
- Si uso Bootstrap, lo trataría como base o set de utilidades, no como excusa para llenar el proyecto de overrides difíciles de seguir.

## Qué diría de SASS/LESS
- Son útiles para variables, composición, parciales y ciertas abstracciones repetibles.
- Pero hay que vigilar no esconder complejidad en anidaciones profundas o mixins que generan CSS imposible de rastrear.

## Señales de una base sana
- Nombres consistentes.
- Componentes con superficie visual acotada.
- Uso claro de design tokens.
- Estrategia explícita para responsive, estados y temas.

## Error común
Pensar que usar un preprocesador ya resuelve la arquitectura CSS. Si no hay límites claros, el proyecto acaba con cascada impredecible, especificidad creciente y miedo a tocar estilos.

## Frase corta de entrevista
> Uso SASS o LESS para ordenar y reutilizar, no para esconder complejidad: tokens claros, estilos por componente y muy pocos overrides globales.
