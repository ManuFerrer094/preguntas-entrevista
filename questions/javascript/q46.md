Si el puesto menciona Web Components y accesibilidad, una respuesta fuerte debe cubrir más que snapshots visuales. Lo importante es demostrar que probarías comportamiento, semántica y accesibilidad real del componente, especialmente cuando hay Shadow DOM de por medio.

## Qué evalúa el entrevistador
- Si conoces herramientas habituales del ecosistema como `@web/test-runner` y `axe-core`.
- Si sabes que la accesibilidad no se valida solo con inspección manual del DOM.
- Si piensas en pruebas automáticas y también en validación del árbol de accesibilidad.

## Respuesta sólida
- Probaría primero comportamiento funcional: render, atributos, propiedades, eventos y ciclo de vida.
- Añadiría checks automáticos de accesibilidad con `axe-core` o integraciones de `open-wc/testing`.
- Para casos complejos de Shadow DOM, revisaría el árbol de accesibilidad o snapshots accesibles para confirmar relaciones de nombre, descripción y rol.

## Ejemplo de enfoque

```js
import { fixture, expect } from '@open-wc/testing';

it('es accesible', async () => {
  const el = await fixture('<user-badge status="ok"></user-badge>');
  await expect(el).to.be.accessible();
});
```

Ese test es un muy buen punto de partida, pero no basta para todo. Si el componente depende de labels, descripciones o relaciones semánticas complejas, añadiría pruebas específicas del árbol de accesibilidad y de interacción real.

## Qué más incluiría
- Casos de teclado y foco.
- Estados de error y validación en formularios.
- Revisión de eventos personalizados.
- Pruebas de integración para consumo desde otros frameworks si el componente se distribuye como librería.

## Error común
Confiar solo en snapshots HTML o pruebas visuales. Un Web Component puede verse bien y seguir siendo inaccesible o emitir un contrato roto para quien lo integra.

## Frase corta de entrevista
> Probaría Web Components en tres capas: comportamiento, accesibilidad automática con `axe-core` y validación semántica real cuando el Shadow DOM complica las relaciones accesibles.
