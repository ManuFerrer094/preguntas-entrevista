Los Web Workers permiten ejecutar JavaScript en un hilo separado del principal del navegador. Son útiles para tareas pesadas que no deben bloquear la interfaz.

## Ejemplo de código
```js
// main.js
const worker = new Worker('worker.js');
worker.postMessage({ numeros: [1, 2, 3, 4] });
worker.onmessage = (event) => {
  console.log(event.data);
};
```

```js
// worker.js
self.onmessage = (event) => {
  const suma = event.data.numeros.reduce((acc, n) => acc + n, 0);
  self.postMessage(suma);
};
```

## Qué aportan
- liberan el hilo principal para que la UI siga respondiendo;
- permiten hacer cálculos intensivos en segundo plano;
- reducen bloqueos perceptibles por el usuario.

## Matices importantes
- No comparten memoria por defecto como si fuera el mismo contexto; se comunican mediante mensajes.
- No tienen acceso directo al DOM.
- Introducen coste de serialización, así que no todo merece un worker.

## Para interiorizarlo
Úsalos cuando el problema sea realmente de trabajo CPU-intensivo. Si el cuello de botella es red, render o diseño de datos, un worker no lo arregla por sí solo.
