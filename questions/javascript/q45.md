Esta pregunta busca ver si piensas en capas. Lo fácil es llamar a `fetch()` desde cualquier componente. Lo valioso es diseñar un flujo donde la interfaz no quede atada a URLs, payloads ni detalles accidentales del backend.

## Qué evalúa el entrevistador
- Si separas acceso a datos, transformación y render.
- Si contemplas estados de carga, error, retry y cancelación.
- Si sabes que JSON de API no equivale automáticamente a modelo de dominio o de pantalla.

## Respuesta sólida
- Pondría el acceso REST/JSON en un servicio o repositorio dedicado.
- Adaptaría la respuesta a un modelo que la UI entienda, en vez de pasar el payload crudo por toda la app.
- La vista consumiría estado ya preparado: loading, data, empty, error.

## Ejemplo de enfoque

```js
async function getAccounts(apiClient) {
  const response = await apiClient.get('/accounts');

  return response.items.map((item) => ({
    id: item.id,
    alias: item.display_name,
    balance: Number(item.balance),
  }));
}
```

El componente no debería saber que el backend devuelve `display_name` o que el saldo llega como string. Esa traducción pertenece a la capa de datos o de adaptación.

## Qué más suma en entrevista
- Hablar de `AbortController` para cancelar peticiones obsoletas.
- Mencionar validación básica de respuesta y tratamiento uniforme de errores.
- Explicar política de caché, revalidación o deduplicación si la SPA crece.

## Error común
Mezclar `fetch`, normalización, reglas de negocio y render en el mismo archivo del componente. Eso vuelve la UI rígida y hace muy costoso cambiar backend, testear o reutilizar lógica.

## Frase corta de entrevista
> La vista no debería hablar REST; debería consumir estado listo para render. El transporte y la adaptación viven en una capa separada.
