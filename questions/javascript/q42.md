Esta es una muy buena pregunta para separar conocimiento superficial de experiencia real con Web Components. El Shadow DOM da encapsulación, pero ciertas relaciones accesibles que en Light DOM son triviales dejan de funcionar igual y obligan a diseñar con intención.

## Qué evalúa el entrevistador
- Si sabes que referencias por ID como `label[for]`, `aria-labelledby` o `aria-describedby` no atraviesan límites de shadow sin más.
- Si puedes proponer patrones concretos para preservar la accesibilidad.
- Si entiendes que una API bonita pero inaccesible no es aceptable en un sistema de diseño serio.

## Respuesta sólida
- El problema típico es asumir que una relación semántica basada en IDs funcionará igual dentro y fuera del Shadow DOM.
- Para resolverlo, diseñaría el componente con una estrategia explícita: wrappers, proxies, slots bien estructurados, delegación de ARIA o elementos espejo que mantengan la relación accesible donde sí puede construirse.
- También validaría el resultado en el árbol de accesibilidad, no solo en el DOM.

## Patrones útiles
- `Wrapper`: el componente envuelve contenido proyectado y mantiene internamente la relación accesible.
- `Decorator`: el componente añade semántica o atributos sobre nodos controlados.
- `Emitter` o proxy: traslada información accesible desde el host al control real.
- `ElementInternals` puede ayudar en ciertos casos, por ejemplo con estados de formularios y semántica del host.

## Cómo lo aterrizaría
Si construyo un campo con label, help text y error message, no confiaría en referencias por ID cruzando shadow boundaries. Preferiría controlar la composición o exponer una API clara para que el componente gestione esas asociaciones de forma segura.

## Error común
Pensar que pasar un `aria-describedby="algo"` desde fuera basta siempre. En Shadow DOM eso puede ser falso. Lo correcto es verificar qué recibe realmente el usuario asistido y qué ve el árbol de accesibilidad.

## Frase corta de entrevista
> En Shadow DOM la accesibilidad no se improvisa: las relaciones por ID no siempre cruzan el límite, así que el componente debe diseñarse para reconstruir esa semántica de forma explícita.
