Esta pregunta va más allá de saber que el Shadow DOM existe. Lo que suelen buscar es criterio: cuándo la encapsulación ayuda de verdad y cuándo se convierte en una barrera para estilos, accesibilidad o integración con el resto del sistema.

## Qué evalúa el entrevistador
- Si sabes qué aísla realmente el Shadow DOM.
- Si puedes explicar el coste de esa encapsulación en tematización, testeo o relaciones semánticas.
- Si distingues entre componentes base muy reutilizables y componentes que viven mejor en Light DOM.

## Respuesta sólida
- Usaría `Shadow DOM` cuando necesito encapsulación real de estilos y estructura interna estable.
- Es especialmente útil en librerías compartidas, sistemas de diseño y componentes que deben sobrevivir a CSS global agresivo.
- Preferiría `Light DOM` cuando necesito que el contenido participe de forma natural en el árbol del documento, herede estilos globales o se integre con semántica y tooling ya existentes sin capas extra.

## Qué ganamos con Shadow DOM
- Aislamiento de CSS.
- Menor riesgo de colisiones de selectores.
- API pública más clara: expongo slots, partes y propiedades, no el marcado interno completo.

## Qué sacrificamos
- Algunas relaciones por ID y ciertos patrones de accesibilidad requieren soluciones explícitas.
- La tematización debe pensarse con `::part`, custom properties o slots.
- Las herramientas y los tests tienen que contemplar límites de shadow.

## Criterio práctico
En una entidad bancaria o en un portal con muchos equipos, el Shadow DOM ayuda mucho cuando el valor del componente está en ser robusto y portable. Pero si el componente es más una mejora progresiva sobre HTML abierto y la integración semántica es prioritaria, Light DOM puede ser mejor decisión.

## Error común
Responder "siempre Shadow DOM porque aísla". La buena respuesta reconoce que la encapsulación es poderosa, pero no gratis. Si complica demasiado accesibilidad, theming o contenido proyectado, quizá no era el contenedor adecuado.

## Frase corta de entrevista
> Uso Shadow DOM cuando quiero un contrato fuerte y resistente al CSS externo; uso Light DOM cuando la integración natural con el documento importa más que el aislamiento.
