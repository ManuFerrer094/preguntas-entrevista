---
title: ¿Qué es el Garbage Collector y cómo funcionan las generaciones en .NET?
difficulty: medium

El **Garbage Collector (GC)** es el componente del CLR responsable de **gestionar automáticamente la memoria heap**. Detecta los objetos que ya no tienen referencias accesibles y libera la memoria que ocupan, evitando fugas de memoria manuales.

**Modelo de generaciones:**

El GC divide el heap en tres generaciones para optimizar la recolección:

| Generación | Contenido | Frecuencia de recolección |
|---|---|---|
| **Gen 0** | Objetos recién creados | Muy frecuente (más barata) |
| **Gen 1** | Objetos que sobrevivieron una recolección | Moderada |
| **Gen 2** | Objetos de larga vida (estáticos, cachés…) | Poco frecuente (más costosa) |

**Flujo de vida de un objeto:**

1. El objeto se crea en **Gen 0**.
2. Si sobrevive una recolección de Gen 0, asciende a **Gen 1**.
3. Si sobrevive una recolección de Gen 1, asciende a **Gen 2**.
4. Una recolección de **Gen 2** (Full GC) recorre todo el heap; es la más costosa.

**Large Object Heap (LOH):**
Los objetos ≥ 85 000 bytes se asignan directamente en el LOH, que se recolecta junto con Gen 2.

**Ejemplo — forzar una recolección (no recomendado en producción):**

```csharp
// Crear y liberar objetos
var lista = new List<byte[]>();
for (int i = 0; i < 1000; i++)
    lista.Add(new byte[1024]);

lista.Clear();
lista = null;

GC.Collect();          // Fuerza recolección de todas las generaciones
GC.WaitForPendingFinalizers();
Console.WriteLine($"Memoria usada: {GC.GetTotalMemory(false):N0} bytes");
```

**Finalizers e IDisposable:**

```csharp
public class Recurso : IDisposable
{
    private bool _disposed = false;

    public void Dispose()
    {
        if (!_disposed)
        {
            // Liberar recursos no administrados
            _disposed = true;
            GC.SuppressFinalize(this); // Evita que el GC llame al finalizador
        }
    }

    ~Recurso() => Dispose(); // Fallback si se olvida llamar a Dispose
}
```

> **Buena práctica:** implementa `IDisposable` para liberar recursos no administrados (conexiones, streams, handles) de forma determinista y usa bloques `using` para garantizar su liberación.
