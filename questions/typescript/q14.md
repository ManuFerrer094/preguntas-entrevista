---
title: ¿Qué es strict mode en TypeScript?
difficulty: medium
tags: [typescript, strict-mode, tsconfig, best-practices]
---

El modo strict de TypeScript es un conjunto de comprobaciones de tipo más rigurosas que se activan con la opción `"strict": true` en `tsconfig.json`. Es una bandera "paraguas" que activa múltiples comprobaciones individuales a la vez. Se recomienda encarecidamente en todos los proyectos nuevos, ya que detecta muchos errores comunes en tiempo de compilación.

## Opciones incluidas en strict

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    // Equivale a activar todas estas opciones:
    // "strictNullChecks": true,
    // "strictFunctionTypes": true,
    // "strictBindCallApply": true,
    // "strictPropertyInitialization": true,
    // "noImplicitAny": true,
    // "noImplicitThis": true,
    // "alwaysStrict": true,
    // "useUnknownInCatchVariables": true
  }
}
```

## strictNullChecks — la más importante

Sin `strictNullChecks`, `null` y `undefined` son asignables a cualquier tipo (comportamiento de TypeScript por defecto sin strict). Con ella, deben manejarse explícitamente:

```typescript
// Sin strictNullChecks (no recomendado)
let name: string = null; // ✅ Sin error (peligroso)

// Con strictNullChecks
let name: string = null; // ❌ Error: null no es string
let name: string | null = null; // ✅ Debe declararse explícitamente

// Narrowing obligatorio
function greet(name: string | null) {
  // name.toUpperCase(); // ❌ Error: name podría ser null

  // Debe verificarse antes de usar
  if (name !== null) {
    name.toUpperCase(); // ✅ TypeScript sabe que es string
  }

  // O con optional chaining
  const upper = name?.toUpperCase(); // upper: string | undefined
}
```

## noImplicitAny — sin tipado implícito any

```typescript
// Sin noImplicitAny: TypeScript inferiría 'any' silenciosamente
function processData(data) { // ❌ Con strict: parámetro tiene any implícito
  return data.name; // Peligroso: no hay verificación de tipo
}

// Con noImplicitAny: debe declararse el tipo
function processData(data: unknown) { // ✅
  if (typeof data === 'object' && data !== null && 'name' in data) {
    return (data as { name: string }).name;
  }
}
```

## strictPropertyInitialization — propiedades de clase inicializadas

```typescript
// Con strictPropertyInitialization
class UserService {
  // ❌ Error: 'userRepository' no está inicializado en el constructor
  private userRepository: UserRepository;

  // ✅ Opciones para resolver:

  // 1. Inicializar en declaración
  private userRepository: UserRepository = new UserRepository();

  // 2. Inicializar en constructor
  constructor(private userRepository: UserRepository) {}

  // 3. Declarar como potencialmente undefined
  private userRepository: UserRepository | undefined;

  // 4. Usar ! si sabes que se inicializará (con cuidado)
  private userRepository!: UserRepository; // Asegura al compilador que será inicializado
}
```

## useUnknownInCatchVariables (TypeScript 4.4+)

```typescript
// Sin useUnknownInCatchVariables: error es any (peligroso)
try {
  riskyOperation();
} catch (error) {
  error.message; // ❌ Con strict: 'error' es unknown, no se puede acceder
}

// Con strict: error es unknown, hay que verificar
try {
  riskyOperation();
} catch (error) {
  if (error instanceof Error) {
    console.log(error.message); // ✅ TypeScript sabe que es Error
  } else {
    console.log(String(error));
  }
}
```

## Configuración recomendada adicional

```json
{
  "compilerOptions": {
    "strict": true,
    // Opciones adicionales recomendadas:
    "noUncheckedIndexedAccess": true, // T[index] es T | undefined
    "exactOptionalPropertyTypes": true, // { a?: string } !== { a?: string | undefined }
    "noImplicitReturns": true,          // Todas las rutas de código deben retornar
    "noFallthroughCasesInSwitch": true, // No fallthrough en switch sin break
    "forceConsistentCasingInFileNames": true
  }
}
```

**Consejo de entrevista**: El modo strict es el estándar en proyectos TypeScript profesionales. La opción más impactante es `strictNullChecks`, que es la causa de la mayoría de los "errores raros" que los desarrolladores encuentran al migrar código JavaScript a TypeScript o al activar strict en proyectos existentes. Siempre empezar proyectos nuevos con `"strict": true`.
