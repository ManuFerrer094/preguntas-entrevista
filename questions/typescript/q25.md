---
title: ¿Qué son las clases abstractas en TypeScript?

Las **clases abstractas** en TypeScript son clases que **no pueden instanciarse directamente** y sirven como plantillas base para sus subclases. Pueden contener implementaciones concretas (que las subclases heredan) y métodos abstractos (que las subclases deben implementar).

## Definición y uso básico

```typescript
abstract class Animal {
  // Propiedad normal (heredada por subclases)
  protected nombre: string;

  constructor(nombre: string) {
    this.nombre = nombre;
  }

  // Método concreto: tiene implementación, las subclases lo heredan
  presentarse(): string {
    return `Soy ${this.nombre} y hago: ${this.sonido()}`;
  }

  // Método abstracto: sin implementación, las subclases DEBEN implementarlo
  abstract sonido(): string;
  abstract moverse(): string;
}

// ❌ No se puede instanciar una clase abstracta:
// const animal = new Animal('Genérico');  // Error: no se puede crear instancia de clase abstracta

// ✅ Las subclases deben implementar todos los métodos abstractos
class Perro extends Animal {
  sonido(): string { return '¡Guau!'; }
  moverse(): string { return 'Corro en cuatro patas'; }
}

class Aguila extends Animal {
  sonido(): string { return '¡Screech!'; }
  moverse(): string { return 'Vuelo con mis alas'; }
}

const perro = new Perro('Rex');
console.log(perro.presentarse());  // "Soy Rex y hago: ¡Guau!"
```

## Clases abstractas vs Interfaces

```typescript
// Interface: solo declara la forma (no puede tener implementación)
interface IRepositorio<T> {
  obtenerPorId(id: number): Promise<T>;
  guardar(entidad: T): Promise<T>;
  eliminar(id: number): Promise<void>;
}

// Clase abstracta: puede mezclar métodos abstractos e implementados
abstract class RepositorioBase<T> {
  // Implementación concreta reutilizable (DRY)
  protected abstract coleccion: string;

  async obtenerPorId(id: number): Promise<T | null> {
    const resultado = await db.query(
      `SELECT * FROM ${this.coleccion} WHERE id = ?`, [id]
    );
    return resultado[0] ?? null;
  }

  // Cada subclase implementa su propia validación
  abstract validar(entidad: T): boolean;

  async guardar(entidad: T): Promise<T> {
    if (!this.validar(entidad)) {
      throw new Error('Entidad no válida');
    }
    // Implementación común de guardar
    return await db.save(this.coleccion, entidad);
  }

  async eliminar(id: number): Promise<void> {
    await db.query(`DELETE FROM ${this.coleccion} WHERE id = ?`, [id]);
  }
}

// Implementación concreta
class RepositorioUsuarios extends RepositorioBase<Usuario> {
  protected coleccion = 'usuarios';

  validar(usuario: Usuario): boolean {
    return !!usuario.email && usuario.email.includes('@')
        && !!usuario.nombre && usuario.nombre.length >= 2;
  }

  // Método adicional específico de usuarios
  async buscarPorEmail(email: string): Promise<Usuario | null> {
    const resultado = await db.query(
      `SELECT * FROM ${this.coleccion} WHERE email = ?`, [email]
    );
    return resultado[0] ?? null;
  }
}
```

## Patrón Template Method con clase abstracta

```typescript
// El patrón Template Method define el esqueleto de un algoritmo
// y deja que las subclases sobreescriban pasos específicos
abstract class ProcesadorDePago {
  // Template Method: define el flujo completo
  async procesar(monto: number, datos: DatosPago): Promise<ResultadoPago> {
    // Pasos fijos del algoritmo
    this.validarDatos(datos);

    // Pasos que cada proveedor implementa diferente
    const tokenizado = await this.tokenizarTarjeta(datos.tarjeta);
    const resultado = await this.realizarCargo(monto, tokenizado);
    await this.registrarTransaccion(resultado);

    return resultado;
  }

  // Pasos con implementación por defecto (pueden sobreescribirse)
  protected validarDatos(datos: DatosPago): void {
    if (monto <= 0) throw new Error('Monto debe ser positivo');
    if (!datos.tarjeta) throw new Error('Datos de tarjeta requeridos');
  }

  protected async registrarTransaccion(resultado: ResultadoPago): Promise<void> {
    await audit.log('pago', resultado);
  }

  // Pasos abstractos: cada proveedor los implementa
  protected abstract tokenizarTarjeta(tarjeta: DatosTarjeta): Promise<string>;
  protected abstract realizarCargo(monto: number, token: string): Promise<ResultadoPago>;
}

class ProcesadorStripe extends ProcesadorDePago {
  protected async tokenizarTarjeta(tarjeta: DatosTarjeta) {
    return await stripe.tokens.create({ card: tarjeta });
  }

  protected async realizarCargo(monto: number, token: string) {
    return await stripe.charges.create({ amount: monto, source: token });
  }
}

class ProcesadorPayPal extends ProcesadorDePago {
  protected async tokenizarTarjeta(tarjeta: DatosTarjeta) {
    return await paypal.vault.create(tarjeta);
  }

  protected async realizarCargo(monto: number, token: string) {
    return await paypal.payments.create({ amount: monto, funding: token });
  }
}
```

## Cuándo usar clase abstracta vs interfaz

| Criterio                                   | Clase Abstracta | Interfaz   |
|--------------------------------------------|-----------------|------------|
| ¿Tiene implementación por defecto?         | ✅ Sí          | ❌ No      |
| ¿Estado compartido (propiedades)?          | ✅ Sí          | ❌ No      |
| ¿Herencia múltiple?                        | ❌ Una sola    | ✅ Múltiple |
| ¿Define solo la forma del objeto?          | ❌ No solo     | ✅ Sí      |
| ¿Código reutilizable entre subclases?      | ✅ Sí          | ❌ No      |

**Consejo de entrevista**: La regla es: usa **interfaz** cuando solo necesitas definir un contrato (qué hace algo, no cómo lo hace), y usa **clase abstracta** cuando quieres compartir implementación entre subclases relacionadas (patrón Template Method). Un error común es abusar de la herencia de clases abstractas cuando la composición de interfaces sería más flexible. En código moderno TypeScript, se prefiere composición sobre herencia.
