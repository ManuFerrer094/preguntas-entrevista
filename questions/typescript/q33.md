La **Inyección de Dependencias (DI)** es un patrón de diseño donde las dependencias de un objeto se proporcionan externamente en lugar de crearse internamente. En TypeScript, se puede implementar manualmente o usando contenedores IoC como `tsyringe` o `inversify`.

## DI manual con interfaces y funciones

```typescript
// 1. Definir interfaces (contratos)
interface IEmailService {
  enviar(para: string, asunto: string, cuerpo: string): Promise<void>;
}

interface ILogger {
  info(mensaje: string): void;
  error(mensaje: string, error?: Error): void;
}

interface IUsuarioRepositorio {
  obtenerPorId(id: number): Promise<Usuario | null>;
  guardar(usuario: Usuario): Promise<Usuario>;
}

// 2. Implementaciones concretas
class EmailServiceSendGrid implements IEmailService {
  async enviar(para: string, asunto: string, cuerpo: string) {
    // Implementación con SendGrid
  }
}

class ConsoleLogger implements ILogger {
  info(mensaje: string) { console.info(`[INFO] ${mensaje}`); }
  error(mensaje: string, error?: Error) { console.error(`[ERROR] ${mensaje}`, error); }
}

// 3. Servicio que recibe sus dependencias inyectadas
class RegistroUsuarioService {
  constructor(
    private readonly usuarioRepo: IUsuarioRepositorio,
    private readonly emailService: IEmailService,
    private readonly logger: ILogger
  ) {}

  async registrar(datos: DatosRegistro): Promise<Usuario> {
    this.logger.info(`Registrando usuario: ${datos.email}`);

    const existente = await this.usuarioRepo.obtenerPorEmail(datos.email);
    if (existente) throw new Error('El email ya está registrado');

    const usuario = await this.usuarioRepo.guardar({
      ...datos,
      fechaRegistro: new Date()
    });

    await this.emailService.enviar(
      usuario.email,
      '¡Bienvenido!',
      `Hola ${usuario.nombre}, tu cuenta ha sido creada.`
    );

    return usuario;
  }
}

// 4. Composición de dependencias (en el punto de entrada)
const servicio = new RegistroUsuarioService(
  new UsuarioRepositorioMySQL(db),
  new EmailServiceSendGrid(apiKey),
  new ConsoleLogger()
);
```

## Contenedor IoC simple manual

```typescript
type Constructor<T = unknown> = new (...args: any[]) => T;

class Contenedor {
  private registros = new Map<string | symbol, () => unknown>();
  private instancias = new Map<string | symbol, unknown>();

  // Registrar como singleton
  registerSingleton<T>(token: string | symbol, factory: () => T): void {
    this.registros.set(token, factory);
  }

  // Registrar como transient (nueva instancia en cada resolve)
  registerTransient<T>(token: string | symbol, factory: () => T): void {
    this.registros.set(token, factory);
  }

  // Resolver una dependencia
  resolve<T>(token: string | symbol): T {
    if (this.instancias.has(token)) {
      return this.instancias.get(token) as T;
    }

    const factory = this.registros.get(token);
    if (!factory) throw new Error(`Token no registrado: String(token)`);

    const instancia = factory() as T;
    this.instancias.set(token, instancia);  // Caché singleton
    return instancia;
  }
}

// Tokens de inyección
const TOKENS = {
  Logger: Symbol('Logger'),
  EmailService: Symbol('EmailService'),
  UsuarioRepo: Symbol('UsuarioRepo'),
  RegistroService: Symbol('RegistroService'),
} as const;

// Configurar el contenedor
const container = new Contenedor();

container.registerSingleton(TOKENS.Logger, () => new ConsoleLogger());
container.registerSingleton(TOKENS.EmailService,
  () => new EmailServiceSendGrid(process.env.SENDGRID_KEY!)
);
container.registerSingleton(TOKENS.UsuarioRepo,
  () => new UsuarioRepositorioMySQL(db)
);
container.registerSingleton(TOKENS.RegistroService, () =>
  new RegistroUsuarioService(
    container.resolve(TOKENS.UsuarioRepo),
    container.resolve(TOKENS.EmailService),
    container.resolve(TOKENS.Logger)
  )
);

// Usar el contenedor
const registroService = container.resolve<RegistroUsuarioService>(TOKENS.RegistroService);
```

## Con tsyringe (contenedor IoC popular)

```typescript
// main.ts
import 'reflect-metadata';
import { container } from 'tsyringe';

// Registrar implementaciones
container.register<IEmailService>('IEmailService', { useClass: EmailServiceSendGrid });
container.registerSingleton<ILogger>('ILogger', ConsoleLogger);
container.register<IUsuarioRepositorio>('IUsuarioRepositorio', { useClass: UsuarioRepositorioMySQL });

// servicio.ts
import { injectable, inject } from 'tsyringe';

@injectable()
class RegistroUsuarioService {
  constructor(
    @inject('IUsuarioRepositorio') private usuarioRepo: IUsuarioRepositorio,
    @inject('IEmailService') private emailService: IEmailService,
    @inject('ILogger') private logger: ILogger
  ) {}

  // ... métodos
}

// Uso: el contenedor inyecta automáticamente
const servicio = container.resolve(RegistroUsuarioService);
```

## DI para testing: inyectar mocks

```typescript
// En tests: inyectar mocks en lugar de implementaciones reales
describe('RegistroUsuarioService', () => {
  let servicio: RegistroUsuarioService;
  let mockEmailService: jest.Mocked<IEmailService>;
  let mockLogger: jest.Mocked<ILogger>;
  let mockRepo: jest.Mocked<IUsuarioRepositorio>;

  beforeEach(() => {
    // Crear mocks tipados
    mockEmailService = { enviar: jest.fn().mockResolvedValue(undefined) };
    mockLogger = { info: jest.fn(), error: jest.fn() };
    mockRepo = {
      obtenerPorId: jest.fn(),
      obtenerPorEmail: jest.fn().mockResolvedValue(null),
      guardar: jest.fn().mockImplementation(async (u) => ({ ...u, id: 1 }))
    };

    // Inyectar mocks (no implementaciones reales)
    servicio = new RegistroUsuarioService(mockRepo, mockEmailService, mockLogger);
  });

  it('envía email de bienvenida al registrar', async () => {
    await servicio.registrar({ nombre: 'Ana', email: 'ana@test.com', password: '123' });

    expect(mockEmailService.enviar).toHaveBeenCalledWith(
      'ana@test.com',
      '¡Bienvenido!',
      expect.stringContaining('Ana')
    );
  });
});
```

**Consejo de entrevista**: La DI es el patrón de diseño más importante para código testeable y mantenible. La clave es el **Principio de Inversión de Dependencias (D de SOLID)**: depender de abstracciones (interfaces), no de implementaciones concretas. Esto permite sustituir implementaciones en tests, cambiar proveedores (ej: de SendGrid a AWS SES) sin modificar el código consumidor, y reutilizar servicios con diferentes configuraciones. En Angular, el framework gestiona el contenedor DI automáticamente; en Node.js, se usa tsyringe, inversify o un contenedor manual.
