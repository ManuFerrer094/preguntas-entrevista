---
title: ¿Qué es el operador satisfies en TypeScript?
difficulty: medium
tags: [typescript, satisfies, type-checking, inference]
---

El operador `satisfies` (introducido en TypeScript 4.9) valida que un valor cumple un tipo sin forzar el tipo de la variable a ese tipo. Combina la seguridad de la comprobación de tipos con la inferencia del tipo más preciso.

## El problema antes de satisfies

```typescript
type Color = 'red' | 'green' | 'blue' | [number, number, number];

// Con anotación de tipo: TypeScript usa el tipo anotado, no el inferido
const palette: Record<string, Color> = {
  red: [255, 0, 0],
  green: '#00ff00',  // ❌ Error: string no es assignable a Color... espera, sí lo es
  blue: [0, 0, 255]
};

// Problema: TypeScript infiere palette.red como Color, no como [number, number, number]
// No podemos usar métodos de array en palette.red
palette.red.map(x => x);  // ❌ Error: Property 'map' does not exist on type 'Color'
```

## satisfies: validar sin ampliar el tipo

```typescript
type Color = 'red' | 'green' | 'blue' | [number, number, number];

// satisfies valida que cumple Record<string, Color>
// pero infiere el tipo preciso de cada propiedad
const palette = {
  red: [255, 0, 0],
  green: '#00ff00',
  blue: [0, 0, 255]
} satisfies Record<string, Color>;

// ✅ Ahora TypeScript sabe que red es [number, number, number]
palette.red.map(x => x * 2);  // ✅ Funciona
palette.green.toUpperCase();   // ✅ Funciona — green es string

// Y también valida el tipo:
const badPalette = {
  red: 'not-a-color'  // ❌ Error: 'not-a-color' no es asignable a Color
} satisfies Record<string, Color>;
```

## Comparación con anotación de tipo

```typescript
interface Config {
  host: string;
  port: number;
  options?: {
    timeout: number;
    retry: boolean;
  };
}

// Con anotación: tipo es Config, perdemos información precisa
const config1: Config = {
  host: 'localhost',
  port: 3000,
  options: { timeout: 5000, retry: true }
};
// config1.options puede ser undefined según el tipo

// Con satisfies: tipo es el literal, options nunca es undefined aquí
const config2 = {
  host: 'localhost',
  port: 3000,
  options: { timeout: 5000, retry: true }
} satisfies Config;

// TypeScript sabe que options no es undefined en este objeto concreto
config2.options.timeout;  // ✅ Sin necesidad de optional chaining
config2.options?.timeout; // También válido pero innecesario aquí
```

## Caso de uso: objetos de rutas/configuración

```typescript
type Route = {
  path: string;
  component: React.ComponentType;
  auth?: boolean;
};

// satisfies valida cada ruta y a la vez mantiene el tipo literal
const routes = {
  home: { path: '/', component: HomePage },
  login: { path: '/login', component: LoginPage, auth: false },
  dashboard: { path: '/dashboard', component: Dashboard, auth: true },
} satisfies Record<string, Route>;

// Autocomplete en las keys
type RouteName = keyof typeof routes; // 'home' | 'login' | 'dashboard'

// El tipo de cada ruta es preciso (no genérico Route)
const homePath = routes.home.path;  // string (no Route['path'])
```

## Validar enums y constantes

```typescript
type Direction = 'north' | 'south' | 'east' | 'west';

const DIRECTIONS = ['north', 'south', 'east', 'west'] satisfies Direction[];
// El tipo inferido es ('north' | 'south' | 'east' | 'west')[]
// No el amplio string[]

// Validar objeto de configuración
const ENDPOINTS = {
  users: '/api/users',
  posts: '/api/posts',
  comments: '/api/comments',
} satisfies Record<string, `/${string}`>;  // Paths que comienzan con /

ENDPOINTS.users;  // '/api/users' — tipo literal preservado
```

## satisfies con interfaces complejas

```typescript
interface UserPermissions {
  read: boolean;
  write: boolean;
  delete: boolean;
  admin?: boolean;
}

// El objeto cumple la interfaz y preserva los tipos exactos
const adminPermissions = {
  read: true,
  write: true,
  delete: true,
  admin: true  // TypeScript sabe que es exactamente true, no boolean
} satisfies UserPermissions;

// adminPermissions.admin es true, no boolean | undefined
if (adminPermissions.admin) {  // TypeScript sabe que siempre es true aquí
  console.log('Es admin');
}
```

**Consejo de entrevista**: `satisfies` resuelve el clásico dilema entre anotar el tipo (pierdes precisión) y no anotarlo (no hay validación). La regla: usa `satisfies` cuando quieres **validar** que un objeto cumple un tipo pero también quieres **preservar** el tipo más preciso inferido. Es especialmente útil en objetos de configuración, tablas de rutas, y mappings donde cada propiedad puede tener un tipo diferente dentro de una unión.
