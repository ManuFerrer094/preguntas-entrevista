**React Testing Library (RTL)** es la librería estándar para testear componentes React. Su filosofía es testear el comportamiento desde la perspectiva del usuario, no los detalles de implementación, haciendo los tests más robustos ante refactorizaciones.

## Filosofía: "Testea lo que el usuario ve y hace"

```jsx
// ❌ Anti-patrón (Enzyme style): testear implementación interna
expect(wrapper.state('contador')).toBe(1);
expect(wrapper.instance().handleClick).toBeDefined();

// ✅ RTL: testear lo que el usuario ve
expect(screen.getByText('Contador: 1')).toBeInTheDocument();
await userEvent.click(screen.getByRole('button', { name: 'Incrementar' }));
```

## Configuración básica

```bash
# Create React App y Vite ya incluyen RTL
npm install --save-dev @testing-library/react @testing-library/user-event
npm install --save-dev @testing-library/jest-dom   # matchers adicionales
```

## Ejemplo completo: Formulario de login

```jsx
// LoginForm.jsx
export function LoginForm({ onLogin }) {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [error, setError] = useState('');

  async function handleSubmit(e) {
    e.preventDefault();
    try {
      await onLogin(email, password);
    } catch (err) {
      setError('Credenciales incorrectas');
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <label htmlFor="email">Email</label>
      <input id="email" type="email" value={email} onChange={e => setEmail(e.target.value)} />

      <label htmlFor="password">Contraseña</label>
      <input id="password" type="password" value={password} onChange={e => setPassword(e.target.value)} />

      {error && <p role="alert">{error}</p>}

      <button type="submit">Iniciar sesión</button>
    </form>
  );
}
```

```jsx
// LoginForm.test.jsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { LoginForm } from './LoginForm';

describe('LoginForm', () => {
  it('renderiza el formulario correctamente', () => {
    render(<LoginForm onLogin={vi.fn()} />);

    expect(screen.getByLabelText('Email')).toBeInTheDocument();
    expect(screen.getByLabelText('Contraseña')).toBeInTheDocument();
    expect(screen.getByRole('button', { name: 'Iniciar sesión' })).toBeInTheDocument();
  });

  it('llama a onLogin con email y contraseña correctos', async () => {
    const user = userEvent.setup();
    const mockLogin = vi.fn().mockResolvedValue({ ok: true });

    render(<LoginForm onLogin={mockLogin} />);

    await user.type(screen.getByLabelText('Email'), 'ana@ejemplo.com');
    await user.type(screen.getByLabelText('Contraseña'), 'miPassword123');
    await user.click(screen.getByRole('button', { name: 'Iniciar sesión' }));

    expect(mockLogin).toHaveBeenCalledWith('ana@ejemplo.com', 'miPassword123');
  });

  it('muestra error cuando el login falla', async () => {
    const user = userEvent.setup();
    const mockLogin = vi.fn().mockRejectedValue(new Error('401'));

    render(<LoginForm onLogin={mockLogin} />);

    await user.type(screen.getByLabelText('Email'), 'mal@mail.com');
    await user.type(screen.getByLabelText('Contraseña'), 'incorrect');
    await user.click(screen.getByRole('button', { name: 'Iniciar sesión' }));

    expect(await screen.findByRole('alert')).toHaveTextContent('Credenciales incorrectas');
  });
});
```

## Queries: jerarquía de preferencia

```jsx
// 1ª opción — Accesibilidad (role, label, text, placeholder)
screen.getByRole('button', { name: 'Enviar' });
screen.getByLabelText('Email');
screen.getByText('Bienvenido');
screen.getByPlaceholderText('Introduce tu email');

// 2ª opción — Semántica
screen.getByTitle('Cerrar modal');
screen.getByAltText('Foto de perfil');

// 3ª opción — Test IDs (último recurso)
screen.getByTestId('mi-componente');  // data-testid="mi-componente"

// Variantes de queries:
getBy*    // Lanza error si no existe o hay más de uno → tests síncronos
queryBy*  // Retorna null si no existe → para verificar ausencia
findBy*   // Espera asíncronamente (Promesa) → para operaciones async
getAllBy*, queryAllBy*, findAllBy*  // Para múltiples elementos
```

## Test de componente con fetch (mock de API)

```jsx
// ListaUsuarios.test.jsx
import { render, screen, waitForElementToBeRemoved } from '@testing-library/react';

// Mock global de fetch
global.fetch = vi.fn();

it('muestra la lista de usuarios después de cargar', async () => {
  fetch.mockResolvedValueOnce({
    ok: true,
    json: () => Promise.resolve([
      { id: 1, nombre: 'Ana García' },
      { id: 2, nombre: 'Carlos López' }
    ])
  });

  render(<ListaUsuarios />);

  // Verificar estado de carga
  expect(screen.getByText('Cargando...')).toBeInTheDocument();

  // Esperar a que desaparezca el spinner
  await waitForElementToBeRemoved(() => screen.queryByText('Cargando...'));

  // Verificar que se muestran los usuarios
  expect(screen.getByText('Ana García')).toBeInTheDocument();
  expect(screen.getByText('Carlos López')).toBeInTheDocument();
});
```

## Matchers más útiles de @testing-library/jest-dom

```jsx
expect(element).toBeInTheDocument();
expect(element).toBeVisible();
expect(element).toBeDisabled() / toBeEnabled();
expect(element).toHaveTextContent('Hola mundo');
expect(element).toHaveValue('texto en input');
expect(element).toHaveClass('mi-clase');
expect(element).toHaveFocus();
expect(element).toBeChecked();  // checkbox/radio
expect(element).toHaveAttribute('aria-label', 'cerrar');
```

**Consejo de entrevista**: La regla de oro de RTL es: "Cuanto más se parezca tu test a cómo el usuario usa el componente, más confianza te dará". Preferir `getByRole` sobre `getByTestId` porque obliga a tener HTML semántico y accesible. Usar `userEvent` (v14+) sobre `fireEvent` porque simula eventos reales del navegador (incluyendo focus, keydown, etc.), haciendo los tests más realistas.
