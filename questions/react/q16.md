La reconciliación es el proceso por el cual React determina qué partes del DOM real necesitan actualizarse cuando el estado o las props de un componente cambian. En lugar de actualizar todo el DOM (que es lento), React compara el árbol de componentes anterior con el nuevo y aplica solo los cambios mínimos necesarios.

## Virtual DOM y el algoritmo de diffing

React mantiene una representación en memoria del DOM llamada **Virtual DOM**. Cuando ocurre un cambio:

1. React genera un nuevo Virtual DOM (árbol de objetos JavaScript).
2. Compara el nuevo árbol con el anterior (proceso de **diffing**).
3. Calcula el conjunto mínimo de cambios (patch).
4. Aplica esos cambios al DOM real.

El algoritmo de diffing de React tiene complejidad O(n) gracias a dos heurísticas:

**Heurística 1: Elementos de diferente tipo generan árboles completamente nuevos.**

```jsx
// Si cambia de <div> a <span>, React destruye el subárbol y crea uno nuevo
// Estado antiguo:
<div>
  <Counter />
</div>

// Estado nuevo: React destruye Counter y crea un nuevo Counter
<span>
  <Counter />
</span>
```

**Heurística 2: Las keys permiten identificar elementos en listas.**

```jsx
// ❌ Sin keys o con índices: React puede confundirse al reordenar
{items.map((item, index) => <Item key={index} {...item} />)}

// ✅ Con keys estables: React identifica correctamente cada elemento
{items.map(item => <Item key={item.id} {...item} />)}
```

## React Fiber

Fiber es la reescritura del motor de reconciliación de React (introducida en React 16). Permite:

- **Renderizado interrumpible**: React puede pausar el trabajo de reconciliación y retomarlo más tarde.
- **Priorización**: las actualizaciones urgentes (input del usuario) tienen prioridad sobre las no urgentes.
- **Concurrent Mode**: múltiples versiones del árbol pueden prepararse simultáneamente.

```jsx
import { startTransition, useTransition } from 'react';

function SearchResults() {
  const [isPending, startTransition] = useTransition();
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  const handleSearch = (value) => {
    setQuery(value); // Actualización urgente: se aplica inmediatamente

    startTransition(() => {
      // Actualización no urgente: puede interrumpirse si el usuario sigue escribiendo
      setResults(filterResults(value));
    });
  };

  return (
    <div>
      <input value={query} onChange={e => handleSearch(e.target.value)} />
      {isPending && <Spinner />}
      <ResultsList results={results} />
    </div>
  );
}
```

## Cómo el diffing afecta el código

```jsx
// Problema: montar/desmontar en lugar de actualizar
function App({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <UserGreeting /> : <GuestGreeting />} {/* Reconciliación eficiente */}
    </div>
  );
}

// Problema común: condicionales que crean nuevo árbol
function Parent({ showDetails }) {
  return (
    <div>
      {showDetails && (
        <div> {/* React crea/destruye este div y todo su contenido */}
          <ExpensiveComponent />
        </div>
      )}
    </div>
  );
}

// Mejor: mantener en el árbol pero ocultar con CSS si el montaje es costoso
<div style={{ display: showDetails ? 'block' : 'none' }}>
  <ExpensiveComponent />
</div>
```

**Consejo de entrevista**: Entender la reconciliación explica por qué son importantes las `key` props, por qué cambiar el tipo de elemento desmonta el componente perdiendo su estado, y por qué `useTransition` puede mejorar la percepción de rendimiento. Mencionar React Fiber y Concurrent Mode demuestra un conocimiento profundo del funcionamiento interno de React.
