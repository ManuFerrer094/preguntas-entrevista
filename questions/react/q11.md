En una entrevista senior de React, esta pregunta no se responde definiendo conceptos. Esperan que conviertas "¿Qué criterios usarías para elegir entre Zustand, Redux Toolkit y React Query en una app real?" en una decisión de ingeniería con criterios claros, riesgos explícitos y una forma concreta de validar el resultado alrededor de zustand, react-query.

## Qué evalúa el entrevistador
- Si distingues estado fuente, estado derivado y estado remoto sin duplicarlos.
- Si sabes cuándo una solución simple deja de escalar y cuándo una compleja es over-engineering.

## Respuesta sólida
- Identifica qué datos son la fuente de verdad y qué partes solo deberían derivarse.
- Explica cómo fluyen las actualizaciones, dónde viven los efectos y cómo evitas sincronizaciones manuales.
- Menciona qué observabilidad o tests usarías para detectar inconsistencias de estado pronto.

## Trade-offs o errores comunes
- Duplicar estado entre store, formularios, URL y cache suele crear bugs silenciosos.
- El store global no arregla un mal modelo; solo hace más visible el problema.

## Ejemplo o caso real
En una pantalla con filtros, paginación y edición, fijaría una sola fuente de verdad y derivaría el resto para no pelearme con sincronizaciones. En React, lo aterrizaría en una pieza con foco en zustand, react-query antes de generalizar el patrón.

## Frase corta de entrevista
> Primero reduzco fuentes de verdad; después elijo la herramienta de estado que mejor encaja con ese flujo.
